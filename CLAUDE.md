## Project Overview

**SRI Electronic Invoice API** — Node.js/Express REST API for generating, digitally signing, and submitting electronic invoices (*facturas electrónicas*) to Ecuador's SRI. Full document lifecycle: Generate → Sign → Send → Authorize. PostgreSQL for persistence.

## Architecture

```
Route → Validator → Controller → Service → Model / Builder / Helper
```

**Dependency rule:** each layer calls only the layer below it. Controllers never touch models. Services never touch `req`/`res`.

```
src/routes/        URL definitions + validator chains
src/validators/    express-validator field rules
src/middleware/    asyncHandler, validateRequest, errorHandler, idempotency, authenticate, rate-limit, verify-mailgun-webhook
src/controllers/   thin HTTP handlers — one service call, one response
src/services/      business logic and orchestration
src/services/email/  email provider factory + Mailgun provider + templates
src/models/        PostgreSQL CRUD (parameterised queries only)
src/builders/      XML document construction (builder registry)
src/errors/        AppError → ValidationError / NotFoundError / SriError / ConflictError / QuotaExceededError
helpers/           signer.js (XAdES-BES), access-key-generator.js (Module 11), ride-builder.js (RIDE PDF)
db/migrations/     SQL migration files 001–050
assets/            factura_V2.1.0.xsd + xmldsig-core-schema.xsd
```

## CRITICAL RULES

1. **English only** — all identifiers, file names, table names, column names. Spanish only where SRI mandates it (XML element names: `infoTributaria`, `claveAcceso`, SOAP payloads).
2. **Never string-interpolate SQL** — always use `$1, $2` parameterised placeholders.
3. **Never skip layers** — controller → service → model. Never controller → model.
4. **Never read `process.env` directly** — import from `src/config/index.js`.
5. **Validate before signing** — XSD validation runs before the expensive crypto step.
6. **Always log an ERROR audit event before re-throwing** from a SRI service catch block.
7. **No hard deletes** — set `active = false` or update status instead.
8. **Wrap all async route handlers** in `asyncHandler` — never add try/catch in controllers.

## Key Patterns

**Primary keys:** every table's PK is `UUID PRIMARY KEY DEFAULT uuid_generate_v7()` — never `SERIAL`/`BIGSERIAL`. `uuid_generate_v7()` (migration `000_uuid_v7_function.sql`) is a hand-rolled PL/pgSQL implementation (48-bit millisecond timestamp + random bytes, version/variant bits set per RFC 9562) since native `uuidv7()` only ships in PostgreSQL 18+ and the target version here is 14.x minimum — it requires the `pgcrypto` extension (`gen_random_bytes`), also enabled in that same migration. UUIDv7's time-ordered layout means `ORDER BY id` and "everything after id X" cursor queries (e.g. `GET /v1/notifications?sinceId=`) keep working exactly like they did under `SERIAL`, since Postgres's native `uuid` comparison is byte-wise and the leading bytes are the timestamp. Every FK column matches its target's `UUID` type — there is no integer PK anywhere in the schema. Any new migration creating a table must use this same `DEFAULT uuid_generate_v7()` pattern; no Node-side id generation exists or is needed. See ADR-020.

**Sequential numbers:** `SELECT ... FOR UPDATE` inside an explicit transaction — guarantees no duplicate sequential numbers under concurrent load. See `src/services/sequential.service.js`.

**Certificate storage:** private key PEM stored AES-256-GCM encrypted in `issuers.encrypted_private_key`; certificate PEM stored plaintext in `issuers.certificate_pem`. Decrypted at signing time only. Encryption key lives in `ENCRYPTION_KEY` env var.

**Issuer logo:** stored as `BYTEA` in `issuers.logo` (migration 050; replaces the dropped `logo_path` column). Two upload paths: optional `logo` field on `POST /v1/register` (multipart, validated in `registration.controller.js`), or `PATCH /v1/issuers/:id/logo` for an existing issuer (PNG/JPEG/GIF, max 500 KB). The registration multer instance has a 10 MB **whole-request** `fileSize` limit (it must also accommodate the P12 cert upload), so the 500 KB logo cap cannot be enforced by multer there — `registration.controller.js` checks `logoFile.size` manually and throws `INVALID_FILE_UPLOAD` if exceeded. The dedicated logo-upload route's multer instance has no other file fields, so its `limits: { fileSize: 500 * 1024 }` enforces the cap natively. `ride.service.js` reads `issuer.logo` and passes it to `helpers/ride-builder.js`, which renders it in the top-left header of every RIDE PDF (including email attachments).

**Multi-branch support:** one RUC can have multiple issuer rows with different `(branch_code, issue_point_code)` pairs. When creating a branch, supply `sourceIssuerId` instead of a P12 file — the service copies `encrypted_private_key`, `certificate_pem`, `cert_fingerprint`, `cert_expiry` from the source row. Self-service endpoint is `POST /v1/issuers` (omit `sourceIssuerId` to inherit from the tenant's first existing issuer). Creating a branch does NOT mint a new API key — the tenant's existing key covers every branch via the `X-Issuer-Id` header. New branches inherit the tenant's current environment (sandbox or production).

**Issuer lifecycle management:** `PATCH /v1/issuers/:id` edits `tradeName`/`branchAddress` only — `businessName`, `mainAddress`, and `ruc` stay permanently tied to the RUC registration and have no edit path. `DELETE /v1/issuers/:id` soft-deletes (`active = false`), refusing if it's the tenant's last active issuer (`LAST_ISSUER_CANNOT_BE_REMOVED`) or if `documentModel.existsByIssuerId()` finds any document for it in either the `public` or `sandbox` schema (`ISSUER_HAS_DOCUMENTS`) — an issuer with history can never be reused. `PATCH /v1/issuers/:id/activate` reverses the soft-delete, but re-runs the exact same branch/issue-point plan-limit checks `createBranch` does (via `tenantModel.countIssuePointsByBranch`/`countBranchesByTenantId`), so a tenant can't dodge their tier's cap by deactivating one issuer and reactivating another.

**Manual sequential management:** `GET /v1/issuers/:id/sequentials` reports the current and next counter value per active document type, for both `sandbox` and `production` side by side. Because `sequential_numbers` has RLS (per-issuer, per-schema), the two environments cannot be read in one query — `sequential.service.js`'s `getCounters()` issues two separate `db.queryAsIssuer(..., sandbox)` calls (one per schema) and merges the results in JS. `PATCH /v1/issuers/:id/sequentials/:documentType` (`setNext()`) locks the row with `SELECT ... FOR UPDATE` inside the same transaction as the write (mirroring `getNext()`'s locking), so it cannot race a concurrent `POST /v1/documents` and produce a duplicate sequential; rejects with `SEQUENTIAL_CANNOT_DECREASE` if `nextSequential` does not exceed the current value.

**Tenant model:** `tenants` is the root billing entity. One tenant owns one RUC with one or more branches and issuing points (limited by tier). Fields: `email`, `subscription_tier` (FREE/STARTER/GROWTH/BUSINESS), `status` (PENDING_VERIFICATION/ACTIVE/SUSPENDED), `sandbox` (boolean, default `true`), `document_count`, `document_quota`, `preferred_language` (default `'es'`). Tenants are NOT user accounts — no password, no session. The API key IS the credential. `src/constants/subscription-tiers.js` defines per-tier `documentQuota`, `maxBranches`, `maxIssuePointsPerBranch`, and rate limits. `PATCH /v1/tenants/language` updates the preferred language after registration. `GET /v1/tenants/me` resolves identity for a holder of an API key with no other context (e.g. a third-party app linking an existing account) — it returns `req.tenant` as-is, with no DB call, since `authenticate` middleware has already resolved it from the key. The route also runs `requireMatchingEnvironment` (same check `resolveIssuer` uses) so a key minted for one environment cannot resolve identity while the tenant is active in the other — see `src/middleware/require-matching-environment.js`.

**Email localisation:** outgoing emails are localised using `src/locales/` — a cross-cutting layer shared by email templates and (in future) API responses. `getTranslations(lang)` returns the locale object for the given language code, falling back to `'es'`. Each locale file exports a plain object keyed by domain (`email.verifyEmail.*`, `email.invoiceAuthorized.*`). Templates own HTML structure; locales own strings. `SUPPORTED_LANGUAGES` exported from `src/locales/index.js` is the single source of truth for accepted language codes — used by validators on `POST /v1/register` and `PATCH /v1/tenants/language`. The invoice-authorized email is localised using the **issuer's tenant** `preferred_language` (there is no buyer-language field — `email.service.js` resolves it via `issuer.tenant_id` → `tenantModel.findById`), not the buyer's.

**Self-service registration:** `POST /v1/register` (public) — creates tenant + issuer + sandbox API key in one call. Tenant starts PENDING_VERIFICATION. A verification email is sent (fire-and-forget). The returned API key is shown once. `GET /v1/verify-email?token=xxx` activates the tenant. Unverified tenants can use sandbox but cannot promote to production. Optional `verificationRedirectUrl` field redirects the email link to a frontend page (e.g. `https://app.example.com/verify?token=xxx`) instead of directly to the API — stored on the tenant row and used for all subsequent verification emails including resends. Token TTL is configurable via `VERIFICATION_TOKEN_TTL_HOURS` (default 24h). `POST /v1/resend-verification` enforces a 60-second server-side cooldown (checked against `tenants.verification_email_sent_at`) in addition to the IP rate limit. `register()` is for new accounts only — if the email already exists it always rejects with `409 CONFLICT`, regardless of tenant status, pointing the caller at `POST /v1/recover`; see "Account recovery" below.

**Account recovery:** `POST /v1/recover` (public) — regains API key access when it's been lost. Requires the same P12 cert used at registration; `registration.service.js`'s `recover()` parses the uploaded cert and compares its fingerprint against `issuer.cert_fingerprint` before doing anything else. Deliberately anti-enumeration: an unregistered email, a tenant with no issuer, and a fingerprint mismatch all return the exact same generic `{ ok: true, message: ... }` response (mirrors `resendVerification()`'s existing "don't leak whether email exists" pattern) — only a genuine match, the same ownership bar fresh registration accepts, does anything observable. On a match: the API key for the tenant's **actual current environment** is revoked and reissued synchronously (`tenant.sandbox ? 'sandbox' : 'production'` — a promoted tenant recovers their production key, not a useless sandbox one). **Extra validation (forced re-verification):** the tenant is also durably demoted to `PENDING_VERIFICATION` with a fresh token (`tenantModel.demoteToPendingVerification()`, the `activate()` counterpart) — a matching certificate proves possession of the P12, but not control of the registered email inbox, so recovery now requires clicking a fresh verification link before the account regains `ACTIVE`-gated privileges (branch creation, promotion, starting a subscription, minting additional named keys — see the `tenant.status !== ACTIVE` gates in `issuer.controller.js`, `tenant.service.js`, `subscription.service.js`, `api-key.service.js`). The reissued key still works immediately for sandbox document creation in the meantime, same as any freshly-registered `PENDING_VERIFICATION` tenant. The demotion happens even when `EMAIL_PROVIDER=none` (no way to ever click the link) — recoverable via the existing admin override, `POST /v1/admin/tenants/:id/verify`. The email itself is durably enqueued (see "Async worker: pending_effects outbox") reusing the exact same `GET /v1/verify-email` redemption machinery as `resendVerification()` rather than a parallel mechanism. Suspended accounts are only ever told they're suspended (`403 ACCOUNT_SUSPENDED`) after a matching certificate is presented — a caller with the wrong cert already got the generic response and learns nothing about account status. See ADR-021 for the original design rationale (including why a simpler synchronous flow was chosen over a two-step token/confirm design) and its addendum for the forced-re-verification change.

**Legal documents and tenant acceptance (ADR-018):** Two-layer design — a template store (`agreements`, migrations 057/059) and per-tenant personalized snapshots (`tenant_agreements`, migration 061).

`agreements` stores the Markdown source for `TERMS`/`PRIVACY`/`DPA`. Content comes from `docs/agreements/*.md` read at publish time by default, or from an optional `contentMarkdown` field in the same `POST /v1/admin/agreements` body (`{ documentType, version, contentMarkdown? }`) — passing `contentMarkdown` lets an admin publish an edited version directly from the DB without touching the filesystem or redeploying (the file-read fallback still exists for any document type never edited through the API). Either source goes through the same `stripDraftHeader()` (a no-op for admin-submitted text, which never carries the git draft banner) and `substitutePlaceholders()` calls, so `{{token}}` resolution and the resulting HTML rendering are identical regardless of origin. `GET /v1/admin/agreements/versions/:id` fetches one specific version's full `content_markdown` by its row id — `agreements.id` is a global primary key, not scoped per `document_type`, so no `:type` is needed to disambiguate — letting a UI pull existing content into an editor before resubmitting via `publish()`. Public `GET /v1/agreements` lists current versions; `GET /v1/agreements/:type` serves rendered HTML via `markdown-it` with a disclaimer notice prepended (not stored in content). Supports `{{token}}` placeholder substitution: `{{operador.*}}` (`nombre`/`ruc`/`domicilio`/`email`) and `{{soporte.email}}` are resolved once at **publish time** by `publish()` (same value for every tenant); `{{cliente.razonSocial}}`/`{{cliente.ruc}}` are resolved later, per tenant, at **generation time** by `tenant-agreement.service.js`'s `generateForTenant()`. **Two distinct contact emails, two distinct purposes:** `{{operador.email}}` names the legally responsible party's identity (used only in the DPA's opening "Encargado" clause) and must never be swapped for the support inbox. `{{soporte.email}}` (resolved from `ADMIN_NOTIFICATION_EMAIL`, falling back to `OPERATOR_EMAIL` if unset so a published document never bakes in a blank contact) is for every "write to us" invitation — account termination, "para consultas," exercising data rights — used throughout Terms of Service and Privacy Policy. `agreement.service.js`'s `buildDisclaimer()` (the "questions about this content? contact us before accepting" notice, prepended outside the stored content, not a `{{}}` token) follows the same `ADMIN_NOTIFICATION_EMAIL`-first reasoning independently, since it's evaluated at render time rather than publish time. `buildDisclaimer()` is shared by both `GET /v1/agreements/:type` (public, generic) and `GET /v1/tenants/agreements/:type` (tenant-facing, personalized) — same disclaimer either way. Both routes also wrap their final body (disclaimer + rendered markdown) with `agreementService.wrapDocumentHtml(documentType, body)`, which returns a complete, self-contained `<!DOCTYPE html>` page (serif typography, justified paragraphs, a styled `<h1>`/`<h2>`/`<h3>` hierarchy, a centered "paper" container, a `<title>` tag derived from a `DOCUMENT_TYPE_TITLES` map) — the raw markdown-it fragment on its own has no page chrome and renders as unstyled plain text if opened directly in a browser.

`tenant_agreements` stores one **personalized, immutable** Markdown snapshot per `(tenant_id, document_type, template_version)` — all `{{}}` tokens are resolved at generation time (DPA includes the Client's actual `business_name` and `ruc`). Status: `PENDING` (generated, not yet accepted) or `ACCEPTED`. Generation happens: (a) at registration as a fire-and-forget background call once the issuer row exists; (b) lazily when `GET /v1/tenants/agreements` or `GET /v1/tenants/agreements/:type` is called and the tenant has no row for the current template version — so publishing a new template surfaces automatically without any admin job. `UNIQUE (tenant_id, document_type, template_version)` with `ON CONFLICT DO NOTHING` makes all generation calls idempotent.

`POST /v1/register` requires `termsVersion`, validated against the current `TERMS` template version before P12 parsing (`tenantAgreementService.validateTermsVersion()`) — skipped if nothing is published yet. `POST /v1/tenants/agreements` (`{ termsVersion }`) accepts all PENDING instances at once, recording `accepted_at`/`ip`/`user_agent` on each row. Old ACCEPTED rows survive every re-acceptance — full history is preserved. `tenants.agreement_accepted_at`/`agreement_version` are a denormalized cache only.

`GET /v1/tenants/agreements` returns per-type drift detection (`{ needsAcceptance, outdated: [{ documentType, currentVersion, acceptedVersion, status, url }] }`) — third-party integrators should poll this periodically to know when their tenant needs to re-accept updated documents. `GET /v1/tenants/agreements/history` lists all instances (history); `GET /v1/tenants/agreements/:type` renders the tenant's personalized HTML (the stored snapshot — no substitution at read time, already baked in) with the disclaimer prepended.

`POST /v1/tenants/promote` is **gated** on all three document types being `ACCEPTED` (`hasAllAccepted()` → `403 AGREEMENT_ACCEPTANCE_REQUIRED` if not). Admin override: `POST /v1/admin/tenants/:id/agreements` generates instances for pre-existing tenants. Admin-created tenants skip `termsVersion` entirely. `docs/agreements/*.md` holds the source templates (Spanish, entity info filled in, `{{cliente.*}}` tokens awaiting per-tenant substitution).

**User-facing promotion:** `POST /v1/tenants/promote` (authenticated) — checks tenant is ACTIVE, checks all agreements are `ACCEPTED` (`403 AGREEMENT_ACCEPTANCE_REQUIRED` if not), flips `tenants.sandbox = false`, seeds production sequentials for all issuers × document types, revokes all sandbox API keys, and creates matching production keys (one per revoked sandbox key, same label). If the tenant already has an `ACTIVE` subscription (activated pre-promotion while in sandbox), the billing period is **reset to the promotion date** via `subscriptionService.resetPeriodOnPromotion()` — the paid period should count production usage time, not sandbox testing time. Returns `{ apiKeys: [{ label, apiKey }] }` — all tokens shown once, store immediately. This is one-way. Admin override: `POST /v1/admin/tenants/:id/promote` (skips both ACTIVE status and legal acceptance checks).

**Admin API:** `ADMIN_SECRET` env var (64-char hex) protects all `/v1/admin/*` routes via `src/middleware/authenticate-admin.js` (constant-time comparison) with a 20 req/min IP-based rate limiter. Admin tenant routes: create (status ACTIVE, no verification), list, update tier, update status (activate/suspend), manual verify, promote (override — skips both ACTIVE status and legal acceptance checks), event log. Admin issuer routes: create (requires `tenantId`), list, certificate renewal. Admin key routes: create, revoke. `POST /v1/admin/tenants/:id/api-keys` defaults `environment` to the tenant's *current* active environment (`tenant.sandbox ? 'sandbox' : 'production'`) when the body omits it — it does not hardcode `'sandbox'`, so creating a key for an already-promoted tenant without specifying `environment` yields a production key.

**Tenant-scoped API keys (ADR-013):** API keys belong to a tenant, not an issuer. One tenant can mint multiple named keys (e.g. `frontend-prod`, `erp`, `mobile-app`) via `GET / POST / DELETE /v1/keys`. Each key carries an `environment` column (`'sandbox'` or `'production'`). The `authenticate` middleware sets `req.tenant` + `req.apiKey` + `req.keyHash` and does NOT set `req.issuer` (issuer resolution is delegated to the next middleware) — it no longer rejects SUSPENDED tenants itself; see "Suspension" below.

**Suspension:** a `SUSPENDED` tenant is not blocked outright — `src/middleware/require-not-suspended.js` (403 `ACCOUNT_SUSPENDED`) is applied selectively so a suspended tenant can still see what already exists (their own documents/RIDE/XML, subscription and payment-proof history, account status/agreements/events) while every write, and a couple of specific reads that amount to *using* the Service rather than just viewing it, stay blocked. `issuers.routes.js`, `api-keys.routes.js`, `catalogs.routes.js`, `notifications.routes.js`, and `webhook-endpoints.routes.js` have no such exception — `router.use(requireNotSuspended)` right after `router.use(authenticate)` blocks the whole router, same as before this split existed. `documents.routes.js`, `subscriptions.routes.js`, `tenants.routes.js`, and `payments.routes.js` apply it per-route instead, explicitly on each write (and on `GET /:accessKey/authorize`, excluded from the exception because it makes a live SRI call and can fire the authorization email — not passive viewing). A new router with authenticated routes must explicitly decide which pattern it needs; there is no default.

**Per-request issuer resolution:** every authenticated document-endpoint request must include an `X-Issuer-Id` header. The `resolveIssuer` middleware (mounted on `/v1/documents/*`) fetches the issuer, validates `issuer.tenant_id === req.tenant.id` (403 ISSUER_FORBIDDEN if not), validates `req.apiKey.environment === (issuer.sandbox ? 'sandbox' : 'production')` (401 on mismatch), and sets `req.issuer`. Issuer-management routes (`/v1/issuers/:id/...`) bypass this middleware and instead read the issuer from `:id` in the URL with an inline ownership check.

**Document quota enforcement:** usage tracking is decoupled from `tenants` entirely — `tenant_quotas` (migration 073) holds one row per tenant per period (`period_start`/`period_end`, `document_quota`, `document_count`), with an `is_current` flag mirroring `agreements`' per-type versioning pattern (exactly one current row per tenant, enforced by a partial unique index). At the start of every **production** document creation transaction (`!issuer.sandbox`), `document-creation.service.js` calls `tenantQuotaService.consumeOne(client, issuer.tenant_id)`, which runs `UPDATE tenant_quotas SET document_count = document_count + 1 WHERE tenant_id = $1 AND is_current = true AND document_count < document_quota RETURNING id` atomically inside the caller's transaction. If no row returns, a `QuotaExceededError` (402 QUOTA_EXCEEDED) is thrown and the transaction rolls back. Sandbox documents skip this block entirely — testing against SRI pruebas never consumes a tenant's paid quota. `POST /v1/admin/jobs/quota` (external cron, daily cadence) calls `tenantQuotaService.resetDuePeriods()`, which rolls over every period whose `period_end` has passed — anchored to the OLD `period_end`, never "now" (mirrors `subscription.service.js`'s `addBillingPeriod()` philosophy, Common Mistake #26), sizing the new cap from the tenant's current `subscription_tier`. This cycle is deliberately independent of `subscriptions.billing_interval` — see "Yearly billing" below for why a YEARLY subscriber must still get quota refreshed monthly. Tier changes update the *cap* only (`tenantQuotaService.setCap(tenantId, tier)`, called everywhere `tenantModel.updateTier(id, tier)` is), never `document_count` or the period boundaries. `src/models/api-key.model.js`'s `findByKeyHash` (the query behind `authenticate` middleware and `GET /v1/tenants/me`) `LEFT JOIN`s the tenant's current `tenant_quotas` row — LEFT, not INNER, so a tenant somehow missing a quota row degrades to `documentCount`/`documentQuota: null` rather than breaking authentication.

**Tier-aware rate limiting:** `writeLimiter` and `readLimiter` in `src/middleware/rate-limit.js` read `req.tenant.subscriptionTier` and return the tier's limit dynamically. `adminLimiter` is a fixed 20 req/min IP-based limiter applied to all admin routes.

**Subscription + payment pipeline (ADR-017):** `subscriptions` and `payments` tables track a tenant's paid-tier billing. **A subscription only becomes `ACTIVE` once its linked invoice document is SRI-`AUTHORIZED` — never merely on payment** (deliberate: granting paid access against an unauthorized invoice is exactly the kind of gap this product exists to close for everyone else). Tier selection can happen two ways: tenant-initiated via `POST /v1/subscriptions` (`{ tier, billingInterval }`, gated on `tenant.status === ACTIVE` only — works while still in sandbox, since paid-tier grants don't depend on sandbox status either, see below), or at **promotion** — `POST /v1/tenants/promote` (tenant-facing) still accepts the same optional `tier`/`billingInterval` fields for a tenant who didn't pre-subscribe. Promotion itself (sandbox flip, key rotation) always proceeds immediately regardless of billing state. `tenant.service.js`'s `promote()` checks `subscriptionModel.findActiveOrPendingByTenantId()` first — any subscription not `CANCELLED`/`EXPIRED` (`PENDING_PAYMENT`, `PAYMENT_RECEIVED`, `INVOICE_PROCESSING`, or `ACTIVE`), not just `ACTIVE`: if one already exists (e.g. started pre-promotion via `POST /v1/subscriptions`), any `tier`/`billingInterval` passed to `promote` is ignored entirely and the response surfaces that existing subscription instead of starting a new one; only when no such subscription exists does a supplied `tier` kick off the pipeline in the background. `resetPeriodOnPromotion()` still only fires when that existing subscription's status is already `ACTIVE` — a still-`PENDING_PAYMENT`/`PAYMENT_RECEIVED`/`INVOICE_PROCESSING` subscription has no period yet to reset. Both tenant-facing entry points funnel into the same `subscriptionService.createSubscription()` — `createSubscriptionForTenant()` is just the `POST /v1/subscriptions` wrapper that adds the ACTIVE-tenant-status gate (the admin route `POST /v1/admin/tenants/:id/subscriptions` calls `createSubscription` directly, bypassing that gate, since admin-created tenants are already `ACTIVE` by construction). Flow: subscription created (`PENDING_PAYMENT`) + a `payments` row priced from `TIERS[tier].priceMonthlyUsd`/`priceYearlyUsd` (response includes `bankTransfer` instructions from `config.bankTransfer`, see `src/config/index.js`) → the **tenant** (their own API key, not admin) uploads proof of the SPI transfer via `PATCH /v1/payments/:id/proof` (multipart, `src/routes/payments.routes.js`, ownership-checked against `req.tenant.id` inside `subscriptionService.submitPaymentProof` — accepts 1–5 files per request, see "Payment proof files" below) → moves the payment to `REPORTED` → the **operator** reviews the uploaded files (`GET /v1/admin/payments/:id/proofs` to list them, `GET /v1/admin/payments/:id/proofs/:proofId` to view one, mirrors the RIDE PDF `res.send(buffer)` pattern) and calls `PATCH /v1/admin/payments/:id/review` (`{ decision: "VERIFIED" | "REJECTED", rejectionReasonCode }` — one endpoint, not two, since it's one decision). `rejectionReasonCode` is a **predefined enum** (`src/constants/rejection-reasons.js`: `AMOUNT_MISMATCH`, `TRANSFER_NOT_FOUND`, `WRONG_ACCOUNT`, `ILLEGIBLE_PROOF`, `DUPLICATE_SUBMISSION`, `OTHER`), not free text — **required and validator-whitelisted** when rejecting (400 `INVALID_REJECTION_REASON` otherwise, re-checked at the service layer too) — so a frontend can map the stable code to its own localized message instead of displaying whatever an admin typed. Stored as `payments.rejection_reason_code` (migration 068, replaces the old free-text `rejection_reason` column entirely). A `REJECTED` payment can be re-submitted via the same `PATCH /v1/payments/:id/proof` (adds more files without touching the ones already uploaded, clears `rejection_reason_code`, moves back to `REPORTED`) — `submitPaymentProof` only refuses once a payment is `VERIFIED`. `VERIFIED` moves the subscription to `PAYMENT_RECEIVED` → operator self-bills the invoice through their own issuer, then `PATCH /v1/admin/subscriptions/:id/link-invoice` (`{ accessKey }`) attaches it. `documentModel.findByAccessKey()` searches both `public.documents` and `sandbox.documents` (UNION ALL, same pattern as the Mailgun webhook) — the returned row carries a `sandbox` boolean. **If the document is from sandbox** (operator issued a test invoice): `linkInvoice` first resolves the payment's purpose (any pending `TIER_CHANGE`/`RENEWAL`, else `INITIAL`) — this lookup runs *before* the sandbox/production fork so both paths agree on what's being funded — then hands off to `linkSandboxDocument`, which never stores `subscriptions.initial_invoice_document_id` or `payments.invoice_document_id` (both FKs reference `public.documents` only, and `sandbox.documents` is a fully independent id sequence that can collide with production ids). If the document is already `AUTHORIZED`, it applies whatever's pending directly and **immediately** — INITIAL activation, or a `TIER_CHANGE`/`RENEWAL` (see "Tier and billing-interval changes" above for why sandbox changes are always immediate and full-price) — skipping the authorization webhook that only fires for production documents. This is a real, previously-shipped bug fix: the sandbox branch used to unconditionally treat every sandbox-linked document as an INITIAL activation, silently no-opping any pending `TIER_CHANGE`/`RENEWAL` payment (see Common Mistakes below). **If the document is from production**: normal flow — stores `subscriptions.initial_invoice_document_id`, moves to `INVOICE_PROCESSING`. `linkInvoice` itself checks `document.status === 'AUTHORIZED'` and calls `activateIfLinked()` immediately, synchronously, in the same request — this is the common case, since an admin normally issues+sends+authorizes an invoice before linking it. For the reverse ordering (an admin links a not-yet-authorized invoice), nothing fires per-document anymore — `POST /v1/admin/jobs/subscriptions` periodically scans for subscriptions still at `INVOICE_PROCESSING` whose linked document has since become `AUTHORIZED` (`subscriptionService.applyPendingInvoiceLinks()`, see ADR-022's addendum for why this isn't a RabbitMQ effect fired on every document authorization system-wide) and calls the same `activateIfLinked()`. Either path flips the subscription to `ACTIVE`, stamps `current_period_start/end` (+1 month or +1 year per `billing_interval`) on both the subscription and the funding payment, and calls `tenantModel.updateTier()` to actually grant the tier/quota. `PATCH /v1/admin/tenants/:id/tier` (the direct admin override, bypassing this whole pipeline) now logs a `TIER_CHANGED` tenant event. `GET /v1/subscriptions/me` (tenant-facing) returns the tenant's full subscription history with payments nested (proof files themselves are never inlined — see "Payment proof files" below) — there's no notification when a review/activation happens, so this plus `GET /v1/tenants/me` (for the resulting tier/quota) is how a tenant finds out. The schema deliberately has **no payment-gateway-specific columns** — no vendor references exist until a gateway is actually decided and built; those will land in their own dedicated migration at that point, not speculatively now (see NEXT_STEPS.md #9, blocked on a registered legal entity existing — every compliant card processor requires KYC against an entity, not an individual, regardless of which vendor is eventually picked).

**Payment proof files:** proof-of-transfer files live in their own append-only `payment_proofs` table (migration 069), one row per uploaded file — not columns on `payments` (the original `proof_file`/`proof_filename`/`proof_mime_type` columns from migration 053 were dropped). `PATCH /v1/payments/:id/proof` accepts up to 5 files per request (`multer.array('proof', 5)`, same field name repeated per file — the standard multipart convention for an array); `subscriptionService.submitPaymentProof()` additionally caps the *cumulative* active count per payment at `MAX_ACTIVE_PROOFS_PER_PAYMENT` (10), across every upload attempt combined, rejecting with `400 PROOF_FILE_LIMIT_REACHED` past that — this stops repeated resubmission from growing the file list unboundedly. Nothing is ever overwritten: a resubmission after `REJECTED` only *adds* rows, so every file from every attempt (including rejected ones) stays visible for audit purposes. "Delete" (`DELETE /v1/payments/:id/proofs/:proofId`, tenant-facing, blocked once `VERIFIED`) is a **soft** delete (`payment_proofs.active = false`, never a hard `DELETE` — CLAUDE.md rule #7) — a tenant who deletes a file loses their own access to it (`GET /v1/payments/:id/proofs` and `.../proofs/:proofId` both filter to `active = true` for the tenant), but the admin equivalents (`GET /v1/admin/payments/:id/proofs` list, `.../proofs/:proofId` download) always show/serve every row regardless of `active`, so the audit trail survives tenant-side deletion. `src/models/payment-proof.model.js` owns all of this — `createMany`, `findActiveByPaymentId`/`findAllByPaymentId`, `findByIdAndPaymentId`, `countActiveByPaymentId`, `softDelete`. Uses plain `db.query()`, same precedent as `payments`/`subscriptions` (not issuer-scoped, no RLS). `PATCH /v1/payments/:id/proof` also requires a `referenceNumber` body field (max 50 chars, validated in `payments.routes.js`) — the SPI transfer's own reference/confirmation number from the tenant's bank, distinct from `payment.id`. It's stored per file row (`payment_proofs.reference_number`, migration 071 — nullable in the DB since it postdates the table, required at the API layer for every new submission) rather than on a separate submission-level record, since every file uploaded in the same request shares one value and the table has no other concept of "submission." Surfaced back via `formatPaymentProof()`'s `referenceNumber` field (both the tenant list/download responses and the admin equivalents, since both read through the same presenter) and included in the operator-facing `sendPaymentProofSubmitted` email so the operator can cross-reference it against their bank statement.

**Tier and billing-interval changes:** `POST /v1/subscriptions/change-tier` (`{ tier, billingInterval? }`, tenant-facing) lets a tenant with an `ACTIVE` subscription move to a different paid tier and/or switch cadence (e.g. monthly STARTER → yearly GROWTH, or monthly GROWTH → yearly STARTER as a downgrade). `billingInterval` is optional — omitting it targets the subscription's current interval, so tier-only callers behave exactly as before. No automated billing gateway exists (NEXT_STEPS.md #9), so this rides the same manual proof/review pipeline as initial promotion rather than charging anything automatically. `requestTierChange` picks one of three paths:
- **Same-interval upgrade** (target tier's `priceMonthlyUsd`/`priceYearlyUsd` higher than the current tier's, interval unchanged) takes effect immediately, gated on payment. Prorates the difference by the fraction of the current `current_period_start`–`current_period_end` window remaining, and creates a `payments` row (`purpose: 'TIER_CHANGE'`, `target_tier`) on the **existing** subscription — not a new subscription row, since the subscription itself doesn't change until the upgrade is paid for. The tenant uploads proof via the same `PATCH /v1/payments/:id/proof` used for initial payments (no new upload endpoint). Admin reviews it the same way (`PATCH /v1/admin/payments/:id/review`), then calls the same `PATCH /v1/admin/subscriptions/:id/link-invoice` — `linkInvoice` detects a `VERIFIED`, unlinked `TIER_CHANGE` payment for that subscription and links the invoice to the **payment's** `invoice_document_id` instead of the subscription's (`subscriptions.initial_invoice_document_id` — named for what it is: the invoice that funded the subscription's *original* activation, stays exactly as set then and is never overwritten by a later upgrade). `linkInvoice` applies it the same way `activateIfLinked` is applied (immediately and synchronously if the invoice is already `AUTHORIZED` at link time, otherwise picked up by `applyPendingInvoiceLinks()`'s periodic scan — see above); since `payment.target_billing_interval` is null for this path, it flips `subscriptions.tier` to `target_tier` immediately, calls `tenantModel.updateTier()`, and stamps the payment's `period_start`/`period_end` to match the (unchanged) subscription period — the upgrade takes over the remainder of the same cycle, it does not start a fresh one. If the prorated amount rounds to `$0` (~no time left in the current period), `requestTierChange` skips the payment/proof step entirely and applies the upgrade immediately — a tenant can't submit proof of a $0 transfer, and there's nothing to collect.
- **Same-interval downgrade** (lower price, interval unchanged) is scheduled, not immediate, and needs no payment — the current period is already paid for at the higher tier. `requestTierChange` just sets `subscriptions.pending_tier`; the tenant's tier/quota stay untouched until `current_period_end`.
- **Any billing-interval change** — tier same, up, or down — is always deferred to `current_period_end` and billed at the new tier+interval's **full sticker price, never prorated**: mismatched cadences (monthly vs. yearly) can't be neatly prorated against each other, so the current period simply runs out as already paid for, and the new cadence starts its own fresh, fully-paid period. `requestTierChange` creates a `payments` row (`purpose: 'TIER_CHANGE'`, `target_tier`, `target_billing_interval`) exactly like the upgrade path, but `applyTierChangeIfLinked` checks `payment.target_billing_interval` first — if set, it does **not** flip the tier immediately; instead it calls `subscriptionModel.scheduleDowngrade(id, target_tier, target_billing_interval)` (setting both `pending_tier` and `pending_billing_interval`) so the change waits for `current_period_end` like a free downgrade, just now paid-for in advance.
- `POST /v1/admin/jobs/subscriptions` (external cron, daily cadence is enough — unlike the minute-level notification job) calls `subscriptionService.applyScheduledTierChanges()`, which finds every `ACTIVE` subscription with a `pending_tier` past its `current_period_end` — this now covers both free same-interval downgrades and paid deferred interval changes, since both end up setting `pending_tier`. It flips `tier` (and `billing_interval`, via `applyTierChange`'s optional third argument, if `pending_billing_interval` was set), clears both pending columns, grants the new quota, and **rolls `current_period_start`/`current_period_end` forward using the new interval** (`subscription.pending_billing_interval || subscription.billing_interval` — using the *old* interval here would mis-size the new period once interval can change via a pending change). If the change was a paid interval switch, it also stamps `period_start`/`period_end` onto the funding `TIER_CHANGE` payment for per-cycle history, same as `activateIfLinked`/`applyRenewalIfLinked` do for their own funding payment.
- All three paths reject with `409 NO_ACTIVE_SUBSCRIPTION` if the tenant has no `ACTIVE` subscription, `400 TIER_CHANGE_NO_OP` if both `tier` and the resolved `billingInterval` match the current subscription, `400 INVALID_BILLING_INTERVAL` if `billingInterval` is supplied but not `MONTHLY`/`YEARLY`, and `409 TIER_CHANGE_ALREADY_PENDING` if a change is already scheduled or a payment is already in flight — only one tier/interval change can be outstanding at a time. All log a tenant event (`TIER_CHANGE_REQUESTED` when a payment is created, `TIER_CHANGE_SCHEDULED` for a free downgrade at request time or a paid interval change once it's scheduled after authorization; the eventual flip logs `TIER_CHANGED`, the same event type the direct admin override already uses) — event `detail` includes `fromBillingInterval`/`toBillingInterval` alongside `fromTier`/`toTier` so the tenant event log (see below) shows the full history of both dimensions changing over time.
- **While `tenant.sandbox === true`, none of the above three paths apply** — `requestTierChange` branches into `requestSandboxTierChange` instead, right after the shared guards. Sandbox has no meaningful `current_period_end` to prorate against or defer a change to: `resetPeriodOnPromotion` (called from `tenant.service.js`'s `promote()`) discards whatever period was running in sandbox entirely, resetting `current_period_start`/`end` to the moment of promotion regardless of what came before. So every sandbox tier/interval change is priced at the target plan's **full sticker price, never prorated**, and — once its self-billed invoice authorizes — applies **immediately**, never deferred to a period boundary that's about to be thrown away. A downgrade still owes nothing (that's about "you already paid for this," not period math) but also applies immediately rather than being scheduled — so sandbox `pending_tier`/`pending_billing_interval` are never set at all, which incidentally means there's nothing left for `resetPeriodOnPromotion` to worry about clearing at promotion time.

**Recurring renewals:** every `ACTIVE` subscription's `current_period_end` is a real recurring billing date, not just a one-time activation timestamp — `subscriptionService.processDueRenewals()` (called by the same `POST /v1/admin/jobs/subscriptions` job as the downgrade step above, **always after it** in the same tick — see ordering note below) handles the cycle:
- **Reminder + payment.** `subscriptionModel.findDueForRenewalReminder(7)` finds every `ACTIVE` subscription within 7 days of `current_period_end` with no `pending_tier` (a scheduled downgrade already rolls its own period forward for free, see above — it doesn't need a renewal payment too) and no renewal payment already open for this cycle. For each, `createRenewalReminder()` opens a `payments` row (`purpose: 'RENEWAL'`, priced from the tier's current `priceMonthlyUsd`/`priceYearlyUsd`), logs a `RENEWAL_DUE` tenant event, and fires `notificationService.createSubscriptionRenewalDue()` (new `SUBSCRIPTION_RENEWAL_DUE` notification type, fans out to webhooks) plus a `subscriptionRenewalDueTemplate` email (bank transfer instructions included, same as the initial subscription) — this is how a tenant actually finds out they need to renew, since nothing else would tell them.
- **Completion.** The tenant uploads proof via the same `PATCH /v1/payments/:id/proof`, admin reviews and self-bills the same way, then `PATCH /v1/admin/subscriptions/:id/link-invoice` — `linkInvoice` now also detects a `VERIFIED`, unlinked `RENEWAL` payment (mirrors the `TIER_CHANGE` branch exactly) and links the invoice to the **payment's** `invoice_document_id`. On authorization, the same immediate-if-already-authorized / periodic-scan-otherwise pair applies `applyRenewalIfLinked` (third sibling of `activateIfLinked`/`applyTierChangeIfLinked` — see above) — it extends `current_period_start`/`current_period_end` by one billing interval **anchored to the OLD `current_period_end`, never "now"** (`addBillingPeriod()`, shared with `applyScheduledTierChanges` and the initial `activateIfLinked`), so an early or late admin review can't drift the billing date — periods stay back-to-back with no gap or overlap. Logs `SUBSCRIPTION_RENEWED`.
- **Expiry.** `subscriptionModel.findExpiredPastGrace(7)` finds every still-`ACTIVE` subscription whose `current_period_end` passed more than 7 days ago with no renewal ever completing (a completed renewal always re-stamps `current_period_end` into the future, so a row this far past it genuinely never renewed). `expireSubscription()` downgrades the tenant to FREE, flips `subscriptions.status` to `EXPIRED` (already a valid value since migration 052), logs `SUBSCRIPTION_EXPIRED`, and fires a matching notification + email — a tenant should never be silently downgraded with zero communication.
- **Ordering matters:** `applyScheduledTierChanges()` must run before `processDueRenewals()` in the same job invocation (`admin.controller.js`'s `runSubscriptionJobs` does this) — a downgrade that just got applied also just rolled its period into the future, so by the time the expiry check runs, it correctly no longer looks past-grace. Reversing the order would cause every subscription downgrading today to also be flagged as freshly expired.
- This is a manual-pipeline-only concept — once a real payment gateway (NEXT_STEPS.md #9, no vendor chosen yet) exists, it owns the recurring charge schedule natively and this cron-driven reminder/grace/expire cycle becomes gateway-specific automation territory, not a redesign.

**Payment review notifications:** `subscriptionService.reviewPayment()` moves `subscriptions.status` to `PAYMENT_RECEIVED` on a `VERIFIED` decision **only when `payment.purpose === 'INITIAL'`** — that transition is a step in the INITIAL activation flow only; a `TIER_CHANGE`/`RENEWAL` payment's subscription is already `ACTIVE` by the time its payment is reviewed and must stay that way (see Common Mistakes #36). It then fires two more side effects after recording the decision, covering every payment purpose (`INITIAL`, `TIER_CHANGE`, `RENEWAL`) uniformly — only the wording adapts: `notificationService.createPaymentReviewed()` (new `PAYMENT_VERIFIED`/`PAYMENT_REJECTED` notification types, INFO/WARNING severity, fans out to webhooks) and `emailService.sendPaymentReviewed()` (tenant-facing, localized, includes the rejection reason when rejected — resolved from `payment.rejection_reason_code` via each locale's `rejectionReasonLabels` map, not the raw code). `notificationService.createPaymentReviewed()` does the same code→label mapping independently, via its own hardcoded English `REJECTION_REASON_LABELS` (no locale infrastructure in `notification.service.js`) — `metadata.rejectionReasonCode` on the notification carries the raw code for consumers that want to do their own mapping (e.g. a webhook receiver localizing to the tenant's UI language). All three payment-display surfaces (`payment-reviewed.js`, `payment-proof-submitted.js`, and `notification.service.js`'s `metadata`) show `payment.target_tier || subscription.tier`, `payment.target_billing_interval || subscription.billing_interval`, and `payment.total_amount` — never the subscription's own (stale, for a `TIER_CHANGE` payment) tier/interval, and never `payment.amount` (the pre-IVA base) — see Common Mistakes #28. Separately, `submitPaymentProof()` fires `emailService.sendPaymentProofSubmitted()` — an **operator**-facing email (not a `notifications` row; the operator isn't a tenant, there's no webhook/preference concept for them) telling whoever holds `ADMIN_NOTIFICATION_EMAIL` that a proof is waiting for review. That env var is optional — unset means the email is silently skipped (mirrors the `SENTRY_DSN` optional pattern), since it's an operational convenience, not something tenant-facing behavior depends on.

**Yearly billing:** `TIERS[tier].priceYearlyUsd` = `priceMonthlyUsd × 10` (2 months free, the standard SaaS discount convention) for STARTER/GROWTH/BUSINESS. `GET /v1/tiers` (public, no auth) exposes the full catalog — quota, both prices, overage rate, document types, limits — for a future pricing page to read from instead of hardcoding the same numbers a second time. **`documentQuota` is always a monthly figure, regardless of `billingInterval`** — a YEARLY subscriber pays once for the year (at a discount) but is still meant to get the published quota *per month*, not per year. `billing_interval` only changes payment cadence/amount (`current_period_end` is +1 year instead of +1 month) and has no relationship to how often document usage resets: quota now resets on its own independent monthly clock (`tenant_quotas.period_start`/`period_end`, see "Document quota enforcement" above) regardless of `billing_interval` — a yearly payer renewing the subscription once a year still gets their usage counter reset monthly, not yearly. Overage billing (letting a tenant opt into exceeding quota and paying the difference, `overagePerDocumentUsd`) is still unbuilt and blocked on a payment gateway — see NEXT_STEPS.md #10.

**Certificate parsing:** extracted to `src/services/certificate.service.js` — used by both `registration.service.js` and `admin.service.js`.

**XSD validation:** `xmllint` CLI via `execFileSync` against `assets/factura_V2.1.0.xsd`. Must be pre-validation (before signing). `xmllint` must be installed on the server.

**Retry logic:** `fetchWithRetry` in `sri.service.js` — retries only on `fetch` throws (network), never on HTTP-level SRI responses.

**Async worker: pending_effects outbox (ADR-019 Phase 1 / ADR-022 Phase 2):** `POST /:key/send` and `GET /:key/authorize` never call SRI synchronously — both always queue and return `202`; there is no sync fallback. **Postgres is the only source of truth; RabbitMQ is purely a confirmed-dispatch signal, never the record.** Phase 1 (ADR-019) tracked dispatch directly on `documents` (`send_dispatch_attempted_at`/`authorize_dispatch_attempted_at`). Phase 2 (ADR-022) replaced that with one generic outbox table, `pending_effects` (migration 075), and moved every other async side effect that used to be an unawaited fire-and-forget promise (notifications, webhook fan-out, subscription hooks, transactional emails) onto the same mechanism — 14 effect types total, SRI send/authorize included. (Subscription activation/tier-change/renewal reconciliation deliberately does **not** use this mechanism — see the "Not one of the 14" note below the catalogue.)

A new `PENDING_SEND` status still sits between `SIGNED` and `RECEIVED`/`RETURNED` on `documents` (`documents.status` CHECK constraint and the DB-level `enforce_document_state_transition()` trigger — migration 027, updated by migration 074 alongside the JS state machine; the DB trigger hardcodes the same transition graph independently and does not derive it from JS) — this part is unchanged by Phase 2; only *how dispatch is tracked* moved off `documents` and onto `pending_effects`.

`pending_effects` columns: `tenant_id` (NOT NULL, FK — every effect type has exactly one owning tenant available at enqueue time; unlike `notifications`/`tenant_events`/`webhook_deliveries`, which added their `tenant_id` from day one, this table has it from the start too, added deliberately so admin tooling can filter "everything pending for tenant X" without parsing per-type JSONB shapes), `effect_type` (14 values, CHECK-constrained), `payload` (JSONB, ids only — handlers re-fetch fresh rows, never trust serialized data), `dedup_key` (nullable; only `SRI_AUTHORIZE` uses it), `status` (`PENDING → DISPATCHED → DONE`, or `→ FAILED` once `attempt_count` reaches `config.pendingEffects.maxAttempts`, default 5), `dispatch_attempted_at`, `attempt_count`, `last_error`. Never hard-deleted (rule #7) — `DONE`/`FAILED` rows are the audit trail. Not issuer-scoped, no RLS — same precedent as `notifications`/`tenant_events`/`webhook_deliveries`.

Every producer call site follows the same shape: `const effect = await pendingEffectService.enqueue(EffectTypes.X, tenantId, payload); pendingEffectService.dispatch(effect);` (`src/services/pending-effect.service.js`). `enqueue()` is a durable, awaited insert — the effect intent survives a crash even if `dispatch()` never runs. `dispatch()` attempts a broker-confirmed publish (`publishConfirmed`, amqplib confirm channel, 3s timeout) and is best-effort — a failed/timed-out publish never fails the caller's request, mirroring the original `queueSend` behavior; the row is simply left for reconciliation.

Three RabbitMQ queues, not one — `sri.send`, `sri.authorize` (unchanged names/topology from Phase 1), and `app.effects` (new, carries the other 12 types). `src/constants/effect-types.js`'s `routingKeyForEffectType()` maps `SRI_SEND`→`send`, `SRI_AUTHORIZE`→`authorize`, everything else→`effects`. This split matters: `channel.prefetch(n)` is per-*consumer* in RabbitMQ despite being set on the channel, so a burst of slow `WEBHOOK_FANOUT` deliveries (10s HTTP timeout each) can never delay `sri.send`/`sri.authorize` message delivery — merging everything into one queue would have put the highest-stakes part of the pipeline behind side-effect latency.

`workers/worker.js` (renamed from `workers/sri-worker.js` in Phase 2 — the role broadened past SRI-only) is the single Render Background Worker process (`npm run worker`) consuming all three queues on one shared confirm channel, three independent `channel.consume()` registrations. Every message is just `{ effectId }`; `pendingEffectService.process(effectId)` does the actual work identically regardless of which queue delivered it:
- `SELECT ... FOR UPDATE` claims the row — this is what makes RabbitMQ's at-least-once redelivery safe: a duplicate delivery blocks until the first attempt's transaction resolves, then sees the row already `DONE`/`FAILED` and no-ops.
- Looks up the handler via `src/effects/index.js`'s registry (mirrors the `src/builders/index.js` "registry keyed by a type string" idiom — `getHandler(effectType)` is the `getBuilder(documentTypeCode)` equivalent).
- Handler resolves normally → row marked `DONE`.
- Handler resolves with `{ requeue: true }` → row left exactly as-is (not `DONE`, `attempt_count` untouched). **Only `SRI_AUTHORIZE`'s handler ever does this** — SRI's authorization can take time, and "still processing" is neither success nor failure; it just needs checking again later, indefinitely, until SRI decides. `checkAuthorization()`'s `result.pending` branch returns this instead of the document.
- Handler throws a `400 AppError` (a state-machine violation — another delivery already advanced the document past this state, expected under at-least-once redelivery) → treated as benign, marked `DONE` anyway (same `isBenignStateError` reasoning Phase 1 already used).
- Handler throws anything else → `attempt_count++`, `last_error` recorded, `status` flips to `FAILED` once `attempt_count >= maxAttempts`, otherwise left as-is for reconciliation to retry; the error is rethrown so the worker `nack`s the message (`requeue: false` — RabbitMQ never retries a message itself, only Postgres-driven reconciliation does).

`SRI_SEND`/`SRI_AUTHORIZE` are one-shot like every other type, with one wrinkle: `sendToSri` (the `SRI_SEND` handler) durably enqueues an `SRI_AUTHORIZE` row the instant a document becomes `RECEIVED` — but does **not** dispatch it yet, guaranteeing every `RECEIVED` document eventually gets an authorize-check even if the client never calls `GET /:key/authorize`. `queueAuthorizationCheck` (the HTTP-facing side, called from `documents.controller.js`) finds-or-creates that same row via `dedup_key = 'sri-authorize:'+documentId` (the one effect type that needs dedup — a document's own state machine already prevents `SRI_SEND` duplicates, and the other 12 types are each created from a single call site per real-world event) and dispatches it immediately, skipping the delay a client's explicit request shouldn't wait through.

**Reconciliation** (`queue-reconciliation.service.js`'s `reconcilePendingEffects()`, `POST /v1/admin/jobs/queue-reconciliation`, same external-cron pattern as the other `/v1/admin/jobs/*` endpoints, recommended hourly): one `SELECT ... FOR UPDATE SKIP LOCKED` sweep against `pending_effects` — no more `public`/`sandbox` schema split, since `pending_effects` isn't schema-scoped (this replaces what used to be up to 4 query executions — 2 functions × 2 schemas — with 1). It **re-publishes a fresh message, never runs a handler itself** — keeping "do the work" a single responsibility of the worker so a fallback processor can't drift out of sync with it. Two timing regimes, both evaluated in the one query:

| What needs dispatch | Condition | Config |
|---|---|---|
| `SRI_AUTHORIZE` rows | first attempt: `created_at` older than the delay window; re-attempts: `dispatch_attempted_at` older than the stale window | `authorizeCheckDelayMinutes` / `authorizeStaleMinutes` |
| Every other effect type | `dispatch_attempted_at` NULL or older than the stale window | `effectStaleMinutes` |

**The 14 effect types** (`src/constants/effect-types.js`): `SRI_SEND`, `SRI_AUTHORIZE`, `DOCUMENT_AUTHORIZED_NOTIFICATION`, `INVOICE_AUTHORIZED_EMAIL` (this last pair enqueued together, awaited, from `document-transmission.service.js`'s `checkAuthorization` the moment a document becomes `AUTHORIZED` — the `INVOICE_AUTHORIZED_EMAIL` handler in `src/effects/index.js` also owns the `email_status`/`document_events` bookkeeping that used to be inline in `checkAuthorization` itself), `TENANT_AGREEMENT_GENERATE`/`VERIFICATION_EMAIL_SEND` (from `registration.service.js`'s `register()`/`recover()`/`resendVerification()` — 3 call sites collapse onto 1 `VERIFICATION_EMAIL_SEND` handler), `WEBHOOK_FANOUT` (from `notification.service.js`'s `fireWebhookFanOut`, fired after every notification create/update — `webhookDeliveryService.fanOut()` itself gained a dedup guard against `webhookDeliveryModel.findByNotificationId()` so a redelivered `WEBHOOK_FANOUT` effect can't double-send to an endpoint that already got it), and the subscription/payment lifecycle types `PAYMENT_REVIEWED_NOTIFICATION`/`_EMAIL`, `PAYMENT_PROOF_SUBMITTED_EMAIL`, `SUBSCRIPTION_RENEWAL_DUE_NOTIFICATION`/`_EMAIL`, `SUBSCRIPTION_EXPIRED_NOTIFICATION`/`_EMAIL` (from `subscription.service.js`'s `reviewPayment`/`submitPaymentProof`/`createRenewalReminder`/`expireSubscription` — the `fireAndForget()` helper these used to share is gone, replaced by the same `enqueue`+`dispatch` shape as everywhere else).

**Not one of the 14 — subscription activation/tier-change/renewal reconciliation:** `SUBSCRIPTION_ACTIVATE_IF_LINKED`/`_APPLY_TIER_CHANGE_IF_LINKED`/`_APPLY_RENEWAL_IF_LINKED` were originally planned here, firing unconditionally on every document authorization system-wide and checking internally whether the document happened to be a subscription-funding invoice. Cut before shipping: a real deployment only ever links an already-`AUTHORIZED` invoice (`linkInvoice`'s own immediate-apply check handles that case synchronously, no queue involved), so firing 3 RabbitMQ messages per authorized document *for every tenant* to cover the rare reverse-ordering case wasn't worth it. That rare case — an admin links a not-yet-authorized invoice — is instead caught by `subscriptionService.applyPendingInvoiceLinks()`, a periodic scan run first (before `applyScheduledTierChanges`/`processDueRenewals`) inside `POST /v1/admin/jobs/subscriptions`, mirroring the same job's existing due/expiry scans rather than an event fired from wherever the state might change. See ADR-022's addendum.

`RABBITMQ_URL` is required at startup (`src/config/validate.js`) — without it nothing can ever dispatch. Deployed as a single shared CloudAMQP instance rather than one per system, isolated by vhost/credentials, not by provisioning a separate broker per app.

**Error responses:** all `4xx`/`5xx` responses use RFC 7807 Problem Details (`Content-Type: application/problem+json`) with `type`, `title`, `status`, `code` (stable SCREAMING_SNAKE_CASE i18n key), `detail`, and `instance` (request URL). `AppError` derives `code`/`type`/`title` from the HTTP status automatically; an optional third constructor argument overrides `code` with a domain-specific value (e.g. `new AppError(msg, 400, ErrorCodes.CERTIFICATE_EXPIRED)`). All stable codes live in `src/constants/error-codes.js` — always import from there, never hard-code strings. `ValidationError` and `SriError` subclasses override with domain-specific values (`VALIDATION_FAILED`, `SRI_SUBMISSION_FAILED`). Field-level errors in `ValidationError.errors[]` each carry a `code` derived from the field path with array indices stripped. See ADR-011 and `docs/site/errors/index.md` for the full code catalogue.

**Audit trail:** every lifecycle transition → `document_events` row. Event types: `CREATED`, `SENT`, `STATUS_CHANGED`, `ERROR`, `REBUILT`, `EMAIL_SENT`, `EMAIL_FAILED`, `EMAIL_DELIVERED`, `EMAIL_TEMP_FAILED`, `EMAIL_COMPLAINED`, `EMAIL_SKIPPED` (email intentionally not sent, e.g. no `buyer_email` — distinct from `EMAIL_FAILED`, which means a send was attempted and failed).

**Tenant event log:** `tenant_events` mirrors the document audit trail for tenant-level lifecycle events — registration/verification, and the full subscription/payment/tier-change pipeline (see above). Written by `registration.service.js`, `mailgun-webhook.service.js`, `subscription.service.js`, `tenant.service.js`, and `admin.service.js`. Uses `db.query()` directly — tenants are not issuer-scoped. Adding a new event type requires updating the `chk_tenant_events_event_type` CHECK constraint in a migration. Read back via `GET /v1/tenants/events` (tenant-facing, `tenant.service.js`'s `getEvents()`) and `GET /v1/admin/tenants/:id/events` (admin-facing, `admin.service.js`'s `listTenantEvents()`) — both map rows through the shared `src/presenters/tenant-event.presenter.js` (`{ id, eventType, detail, createdAt }`) and both 404 if the tenant doesn't exist. `tenantEventModel.findByTenantId` existed since migration 036 but had no route reading it back until these two endpoints shipped. `PATCH /v1/admin/tenants/:id/status` (`admin.service.js`'s `updateTenantStatus()`) logs a `STATUS_CHANGED` event (`{ from, to, reason }`, migration 070) — previously silent, so suspending or reactivating a tenant left no audit trail at all, unlike `TIER_CHANGED` on the tier-override endpoint. `reason` (optional, max 200 chars) is what distinguishes *why* a tenant landed in `SUSPENDED` — the product deliberately has no separate "terminated"/"closed" status (see "Suspension" and "Account recovery" above: `SUSPENDED` already blocks writes while letting a tenant keep viewing their own historical documents/payments, which is the correct behavior for a voluntarily closed account too, not just a punitive one) — so `reason` is the only signal that tells a voluntary closure request (`docs/agreements/terms-of-service.md` §10) apart from a fraud/non-payment suspension in the audit trail.

**Builder registry:** `src/builders/index.js` maps document type codes to builder classes. Adding a new document type = new builder + one registry entry. `SUPPORTED_TYPES` (exported from `src/builders/index.js`) is derived from the registry keys and used by validators and `issuer.service.js` to enforce type eligibility. When adding a new builder, `SUPPORTED_TYPES` automatically includes it — no manual update needed.

**Issuer document types:** `issuer_document_types` table records which document types each issuer is allowed to use. Defaults to `['01']` at registration/admin create. Checked at document creation time — attempting to create a disallowed type returns 400. Managed via `GET/POST /v1/issuers/:id/document-types` and `DELETE /v1/issuers/:id/document-types/:code`. At promotion, production sequentials are seeded for all active types (using `initialSequentials` values if provided, otherwise 1). Adding a new document type to the system requires a new builder, not a migration — and adding it to the appropriate tiers' `allowedDocumentTypes` in `src/constants/subscription-tiers.js`, or it ships unreachable on every plan.

**Document type tier gating:** `TIERS[tenant.subscriptionTier].allowedDocumentTypes` (`subscription-tiers.js`) restricts which document types a tenant can newly *enable* — checked in `issuerService.addDocumentType()` and in `createBranch()`'s `documentTypes` field (402 `DOCUMENT_TYPE_NOT_IN_TIER` if not allowed). This only gates adding a new type, never removing one or creating documents of an already-active type — a tenant who downgrades keeps using types already active on their issuers (grandfathered); they just can't activate further gated types until they upgrade again.

**Credit note balance lookup:** `GET /v1/documents/:accessKey/credit-notes` returns the sum of all `AUTHORIZED` credit notes (`document_type = '04'`) already issued against a given document, plus a `remaining` balance, so the caller can enforce "this credit note can't exceed the original's remaining balance." There is no foreign key linking a credit note to the document it credits — `document-creation.service.js` only stores the raw request body in `request_payload` — so the lookup reconstructs the original's own `NNN-NNN-NNNNNNNNN` number from `issuer.branch_code` + `issuer.issue_point_code` + the zero-padded `sequential`, then matches credit notes whose `request_payload->'originalDocument'->>'number'` and `->>'documentType'` equal it (see `documentModel.findCreditNotesByOriginalDocument`). Only `AUTHORIZED` credit notes count toward consumed balance — `SIGNED`/`RECEIVED` (still pending) and `RETURNED`/`NOT_AUTHORIZED` (rejected) are excluded. Deliberately does not hardcode the original document's type as `'01'` — SRI permits other document types to be credited too. The `request_payload` JSONB path query has no supporting index; fine at current volume, but would need one if `documents` grows large. Known race: two credit notes created back-to-back before the first authorizes won't see each other in this sum — accepted, not solved by locking.

**Idempotency key:** `POST /v1/documents` accepts an optional `Idempotency-Key` header. The key and a SHA-256 hash of the request body are stored in `documents.idempotency_key` / `documents.payload_hash`. A duplicate key with the same payload returns the existing document (200). A duplicate key with a different payload throws `ConflictError` (409). Concurrent races are handled by catching `23505` in the transaction rollback path. See `src/middleware/idempotency.js` and ADR-006.

**Email delivery:** when a document becomes `AUTHORIZED`, `emailService.sendInvoiceAuthorized()` is called fire-and-forget. It generates the RIDE PDF and XML on the fly and sends both as attachments via Mailgun. The Mailgun message ID (angle brackets stripped) is stored in `documents.email_message_id`. Per-document status tracked in `documents.email_status` (`PENDING` → `SENT` / `FAILED` / `SKIPPED`). Failed sends retried via `POST /email-retry` (batch) or `POST /:key/email-retry` (single, add `?force=true` to resend an already-sent email). Provider swappable via `EMAIL_PROVIDER` env var + new file in `src/services/email/providers/`. **Sender display name:** invoice/document emails use `"${issuer.business_name} via Comprobify <EMAIL_FROM_DOCUMENTS>"` — the issuer's own business name appears as the sender because the email is sent on behalf of the business that issued the invoice. All other transactional emails (verification, payment reviewed, renewal, expiry) use `"Comprobify <EMAIL_FROM>"` — the display name is hardcoded to "Comprobify" since these are platform-level notifications, not tenant-issued documents. `EMAIL_FROM_DOCUMENTS` is an optional, separate sender address for invoice/document emails only (e.g. `facturas@` vs `notificaciones@`) — it falls back to `EMAIL_FROM` when unset (`config.email.fromDocuments`), so a deployment that only sets `EMAIL_FROM` behaves exactly as before. **Staging banner:** when `config.appEnv === 'staging'`, `email.service.js`'s `applyStagingBanner()` prepends a bold, localized "TEST ENVIRONMENT" / "ENTORNO DE PRUEBAS" notice to every outgoing email's `text` and `html` body, uniformly across all 6 `send*` functions — applied centrally in the service rather than in each template, since template HTML structure isn't consistent (some are full `<html><body>` documents, some are bare fragments; the banner is inserted right after `<body>` when present, otherwise prepended). This mirrors the `ambiente` distinction CLAUDE.md documents for SRI routing, applied to outgoing email. **Do-not-reply notice:** every template's `disclaimer` string in `src/locales/es.js`/`en.js` includes "no responder / do not reply" wording — these are one-way transactional notifications.

**Mailgun webhook:** `POST /v1/mailgun/webhook` receives Mailgun delivery events and updates `email_status`: `delivered` → `DELIVERED`, `failed`+`permanent` → `FAILED`, `failed`+`temporary` → status unchanged + `EMAIL_TEMP_FAILED` event (Mailgun retries), `complained` → `COMPLAINED`. All requests verified with HMAC-SHA256 via `verify-mailgun-webhook.js` middleware (`MAILGUN_WEBHOOK_SIGNING_KEY` config key). Lookup is by `email_message_id` — `findByEmailMessageId` searches **both** `public.documents` and `sandbox.documents` via `UNION ALL` and returns a `sandbox` boolean on the row. That flag is passed to `updateEmailStatus` and `documentEventModel.create` so the update and audit event land in the correct schema. See ADR-010.

**Rate limiting:** per-API-key request rate limits prevent abuse and quota exhaustion. Applied via `src/middleware/rate-limit.js` using `express-rate-limit`: 60 req/min on write endpoints (POST), 300 req/min on read endpoints (GET). Keyed by `req.keyHash` (SHA-256 token hash). Returns RFC 7807 `429 TOO_MANY_REQUESTS` response. Configurable via `RATE_LIMIT_WINDOW_MS` (default: 60000ms) and `RATE_LIMIT_MAX` (default: 60) env vars. See `docs/site/errors/too-many-requests.md` for client retry guidance.

**Sandbox environment + SRI routing:** `APP_ENV` (`staging` | `production`) combined with `tenants.sandbox` (boolean, default `true`) controls which SRI SOAP endpoint is used and what `ambiente` digit is embedded in the access key and XML. `tenants.sandbox` is the **single source of truth** for a tenant's intended SRI environment — `true` means SRI test, `false` means eligible for SRI production (subject to the staging safety rail below). `issuers` no longer has an `environment` column (dropped in migration 049) — `issuer.sandbox` in all service code is a virtual field set by `resolveIssuer` middleware (`req.issuer.sandbox = req.tenant.sandbox`).

| `APP_ENV`    | `tenant.sandbox = true` | `tenant.sandbox = false` |
|---|---|---|
| `staging`    | SRI test, `ambiente = 1` | SRI test, `ambiente = 1` |
| `production` | SRI test, `ambiente = 1` | SRI production, `ambiente = 2` |

`document-creation.service.js` and `document-rebuild.service.js` compute `ambiente = (config.appEnv !== 'production' || issuer.sandbox) ? '1' : '2'` (where `issuer.sandbox` reflects `tenant.sandbox` via the virtual field) and pass an `effectiveIssuer` (with `environment` set to the computed value) to both the builder and `accessKeyService.generate()`. `sri.service.js` `getSriUrls(issuer)` applies the same logic. All tenants default to `sandbox = true` — safe mode until explicitly promoted.

**Sandbox PostgreSQL schema:** sandbox and production documents live in separate schemas (`sandbox` vs `public`) so sequential sequences are fully independent and test data can be truncated safely. The `sandbox` schema contains `documents`, `document_line_items`, `document_events`, `sequential_numbers`, and `sri_responses` with the same constraints, triggers, and RLS as `public`. **Every future migration that alters a tenant-scoped table must be applied to both schemas.**

**`updated_at` maintenance:** there is no ORM — `src/models/` is raw parameterised `pg` SQL — so a `BEFORE UPDATE` trigger (`set_updated_at()`, migration 051) bumps `updated_at = NOW()` on every row update for every table that has the column (`issuers`, `public.documents`, `public.sequential_numbers`, `tenants`, `notification_preferences`, `webhook_endpoints`, `webhook_deliveries`, `sandbox.documents`, `sandbox.sequential_numbers`). This is a DB-level backstop, not a substitute for application code — UPDATE statements may still set `updated_at = NOW()` explicitly (harmless; the trigger overwrites it with the same value) but no longer **must**. A new migration adding an `updated_at` column to a table must also add a matching `trg_<table>_updated_at` trigger in that same migration, or updates to that table will silently never bump it.

**Row-Level Security (RLS):** all issuer-scoped tables (`documents`, `document_line_items`, `document_events`, `sequential_numbers`) have RLS enabled via migration 031. The policy restricts every query to the current issuer by reading `app.current_issuer_id` — a transaction-local PostgreSQL setting. RLS was dropped from `api_keys` in migration 042 because authentication happens before any context can be set; `api_keys` queries filter by `tenant_id` explicitly at the application layer (reintroducing tenant-scoped RLS is noted in `NEXT_STEPS.md`). Two helpers in `src/config/database.js` manage this:
- `db.setIssuerContext(client, issuerId, sandbox)` — call after `BEGIN` on an existing transaction client; sets both `app.current_issuer_id` (for RLS) and `search_path` (`sandbox, public` or `public`), both rolled back automatically on abort.
- `db.queryAsIssuer(issuerId, sql, params, sandbox)` — wraps a single query in a mini BEGIN / set_config / SET LOCAL search_path / query / COMMIT for non-transactional reads.
All authenticated service code paths must use one of these two helpers. Only the Mailgun webhook, admin API, and health check are exempt — they authenticate by other means and operate without an issuer context (the policy's null bypass allows it). The application DB user must **not** be a PostgreSQL superuser; superusers always bypass RLS regardless of policies.

**Notification and webhook system (ADR-015):** tenant-level alerts delivered via webhooks (primary) and polling (fallback). Two creation paths:
- *Event-driven* — `notificationService.createDocumentAuthorized(document, issuer)` runs via a durably-enqueued `DOCUMENT_AUTHORIZED_NOTIFICATION` effect (see "Async worker: pending_effects outbox") from `document-transmission.service.js` when SRI authorises a document. Multiple authorisations within a 60-second window are aggregated into one notification row (same `id`, incrementing `count`). Failure never affects the HTTP response — a failed handler attempt is retried by reconciliation, not swallowed. `createPaymentReviewed(payment, subscription, decision)` (from `subscriptionService.reviewPayment()`) and `createSubscriptionRenewalDue`/`createSubscriptionExpired` (from `subscriptionService.processDueRenewals()`) follow the same durably-enqueued, non-aggregated pattern — see the "Recurring renewals" / "Payment review notifications" entries above.
- *Scheduled* — `notificationService.runCertChecksForTenant(tenantId, prefs)` checks certificate expiry for all tenant issuers and upserts `CERT_EXPIRING`/`CERT_EXPIRED` alerts. Called by `notification-scheduler.service.runAll()`, which is triggered by `POST /v1/admin/jobs/notifications` (called by external cron). Consumers do NOT call any sync endpoint.
After every notification create/update, `webhookDeliveryService.fanOut(notification)` fans the event out to all active, subscribed webhook endpoints (fire-and-forget). Failed deliveries are retried by the admin job. Consumers can also fall back to `GET /v1/notifications?sinceId=<id>` for catch-up polling.
`notifications`, `notification_preferences`, `webhook_endpoints`, and `webhook_deliveries` tables use `db.query()` directly (not issuer-scoped; no RLS). Optional `X-Issuer-Id` filter on `GET /v1/notifications`: parsed by `parseOptionalIssuerId()` in the controller. When supplied, the query adds `AND (issuer_id = $2 OR issuer_id IS NULL)`. Adding a new notification type requires updating the CHECK constraints in both `044_notifications.sql` (or a new migration) and `045_notification_preferences.sql`, plus entries in `NotificationTypes` and `NotificationSeverity` constants.

**Error monitoring (Sentry):** unexpected `5xx` failures are reported to Sentry via `@sentry/node`. `instrument.js` (project root) calls `Sentry.init({ dsn, environment, sendDefaultPii: false })` and is required at the very top of `app.js` — before any other module — so the SDK can auto-instrument `http`, `express`, and `pg`. `Sentry.setupExpressErrorHandler(app)` is mounted in `server.js` immediately before the central `errorHandler`, so it only reports errors with `statusCode >= 500` (or none) and then forwards unchanged; expected `AppError` 4xx responses (validation, not found, quota, etc.) are never sent. `environment` is always `staging` or `production` (mirrors `config.appEnv`), filterable in the Sentry UI. `SENTRY_DSN` is optional — when unset (e.g. local development), the client is a no-op and nothing is transmitted.

`workers/worker.js` also requires `instrument.js`, but Sentry's automatic uncaught-exception capture never fires there — every worker failure path is already caught by the worker's own code (per-message errors in `handleMessage`, consumer-registration failures, startup failures), so nothing is ever technically "uncaught." Two spots call `Sentry.captureException(err)` explicitly instead: the `registerConsumers(...).catch()` inside `start()` (worker is connected but not actually consuming — worse than fully down, since it looks alive) and the top-level `start().catch()` (the worker never came up at all). The latter also `await`s `Sentry.flush(2000)` before `process.exit(1)` — `captureException` only queues the event, it doesn't send it, so exiting immediately after would race the network request and could kill the process before Sentry ever receives it. Per-message failures inside `handleMessage` are deliberately left console-only, not sent to Sentry — they're expected/routine (network hiccups, a slow webhook endpoint, etc., already covered by `queue-reconciliation.service.js`'s re-publish), and reporting every one would just be noise for something the design already handles.

**Config validation:** critical environment variables are validated at startup in `src/config/validate.js` and called from `app.js` before `Server` construction. Always-required: `APP_ENV` (`staging` | `production`), `ENCRYPTION_KEY` (64-char hex format), `ADMIN_SECRET`, `APP_BASE_URL`, `BANK_TRANSFER_BANK_NAME`/`ACCOUNT_TYPE`/`ACCOUNT_NUMBER`/`ACCOUNT_HOLDER`/`IDENTIFICATION` (without these, `POST /v1/subscriptions` hands a tenant an empty bank-transfer block with no way to pay), `ADMIN_NOTIFICATION_EMAIL` (without it, a payment proof submission raises zero notification to anyone), `RABBITMQ_URL` (without it the async send/authorize pipeline has no way to ever dispatch a queued document — see "Async worker: pending_effects outbox" above). Email-required when `EMAIL_PROVIDER` is set to anything other than `'none'`: `MAILGUN_API_KEY`, `MAILGUN_DOMAIN`, `MAILGUN_WEBHOOK_SIGNING_KEY`, `EMAIL_FROM`. If any are missing or malformed, the process throws immediately with a clear error message before accepting any HTTP requests. This prevents silent failures like unsigned webhooks, unencrypted P12 storage, or a subscription pipeline nobody's watching. `IVA_RATE` (default `0.15`, Ecuador's current rate — see `src/constants/subscription-tiers.js`'s comment) is separately configurable but not validated as required, since it has a safe default and Ecuador has changed this rate more than once historically — a rate change should be a config update + restart, never a code change.

## Document Lifecycle

All document endpoints require the `X-Issuer-Id` header (the issuer's UUID, returned by `GET /v1/issuers`).

```
POST /v1/documents             → SIGNED   (Idempotency-Key header optional, documentType required — no default)
POST /:key/send                 → PENDING_SEND (202 — queued; worker later moves it to RECEIVED | RETURNED)
GET  /:key/authorize            → 202 (queued; worker later moves it to AUTHORIZED | NOT_AUTHORIZED  +fires email)
POST /:key/rebuild              → SIGNED  (from RETURNED or NOT_AUTHORIZED)
GET  /:key/ride                 → application/pdf  (AUTHORIZED only)
GET  /:key/xml                  → application/xml  (authorization XML or signed XML)
GET  /:key/events               → audit trail for the document
GET  /:key/sri-responses        → raw SRI reception/authorization call outcomes (operationType, status, messages) for this document
GET  /:key/credit-notes         → sum of AUTHORIZED credit notes issued against this document + remaining balance
POST /email-retry               → batch retry all PENDING/FAILED emails
POST /:key/email-retry          → retry single email (?force=true to resend SENT)
POST /v1/mailgun/webhook       → Mailgun delivery event → update email_status (HMAC-verified)
```

`rebuild` corrects invoice content (taxes, items, buyer, payments) and re-signs using the same `access_key`, `sequential`, and `issue_date`. Used when SRI returns RETURNED or NOT_AUTHORIZED.

**Invoice generation steps:**
1. Idempotency check — if key seen + hash matches, return existing doc immediately
2. Issuer provided via `req.issuer` (set by `authenticate` middleware from API key)
3. `SELECT ... FOR UPDATE` → next sequential
4. Generate 49-digit access key (Module 11 check digit)
5. Build unsigned XML (`InvoiceBuilder`)
6. Validate against XSD (`xmllint`)
7. Sign XML (XAdES-BES, PEM private key + cert from `issuers` row)
8. Read `buyer_email` from `body.buyer.email` (required field)
9. `INSERT` into `documents` (with `idempotency_key`, `payload_hash`, `buyer_email`)
10. `bulkCreate` into `document_line_items`
11. Log `CREATED` event to `document_events`

## Releasing

Every commit on `main` is a merged PR (often squash-merged, so the SHA on `main` differs from any local commit you made on the branch). That means **`npm version`'s built-in commit+tag step cannot run directly on `main`** — it would push a version-bump commit straight to `main`, bypassing review, and the tag would point at a commit that PR review never saw. Release the same way every other change ships, then tag the result:

1. Branch off `main`: `git checkout -b chore/release`
2. Bump the version **without** letting npm create its own commit/tag: `npm --no-git-tag-version version <patch|minor|major>` (updates `package.json` + `package-lock.json` only)
3. In the same branch, rename `CHANGELOG.md`'s `## [Unreleased]` header to `## [X.Y.Z] — <today's date>` (matching the version just written) and start a fresh empty `## [Unreleased]` above it
4. Commit (`chore: bump version to X.Y.Z`), open a PR, merge it like any other change
5. **After** that PR is merged, pull `main`, then tag the resulting merge commit directly — not the commit you made on the branch: `git tag -a vX.Y.Z -m vX.Y.Z && git push origin vX.Y.Z`

The tag still tracks `package.json`'s version — there's just a merge step between bumping it and tagging it, because the squash-merge changes the commit SHA.

Pushing the tag triggers `release-staging.yml`, which fast-forwards `staging` to it — the tag is treated as an **immutable** "build this" snapshot. Never push a follow-up commit to `main` that changes the version after a tag is created; that would leave the tagged commit's `package.json` permanently out of sync with its own tag name, and would race with `staging` already having been fast-forwarded to it. If `package.json`'s version and the latest git tag ever drift apart, fix it with a manual one-off sync commit (`chore:`), then resume this sequence for every release after that.

## Common Mistakes to Avoid

1. Calling a model directly from a controller — always go through the service.
2. String-interpolating SQL — use `$1, $2` always.
3. Throwing `new Error()` instead of an `AppError` subclass — the error handler returns a generic 500.
4. Forgetting `asyncHandler` on a route — async errors are swallowed silently.
5. Forgetting `validateRequest` in the route chain — validator runs but errors are never checked.
6. Signing before XSD validation — signing is expensive; fail fast.
7. Retrying on HTTP-level SRI errors — only retry on `fetch` throws.
8. Not logging an `ERROR` audit event before re-throwing — leaves a gap in the document history.
9. Reading `process.env` directly in a service or model — use `src/config/index.js`.
10. Hardcoding Spanish identifiers in new code — use English everywhere except SRI XML elements.
11. Generating a new idempotency key on every retry — the key must be generated once and reused across retries for the same intended invoice.
12. Adding a new `document_events` event type without updating the `chk_document_events_event_type` CHECK constraint — the INSERT will fail silently if the constraint is not updated in a migration.
13. Calling `db.query()` directly in an authenticated service or model — use `db.queryAsIssuer(issuerId, sql, params, sandbox)` instead so both RLS and the correct `search_path` are set. `db.query()` (no issuer context) is only correct for the webhook, admin, and health code paths. Exception: even in the webhook path, queries against `documents` or `document_events` must use explicit schema prefixes (`public.documents` / `sandbox.documents`) because the webhook has no issuer context to set `search_path` — see `findByEmailMessageId` for the `UNION ALL` pattern.
14. Adding a migration that alters a tenant-scoped table (`documents`, `document_line_items`, `document_events`, `sequential_numbers`, `sri_responses`) without also applying the same DDL to the `sandbox` schema — the schemas must stay structurally identical.
15. Passing `issuer.environment` directly to `accessKeyService.generate()` or the XML builder — always derive `ambiente` from `config.appEnv` and `issuer.sandbox` first, then build an `effectiveIssuer` with the computed value. Using the raw DB field bypasses the staging safety rail and could embed `ambiente = 2` in a document sent to the test endpoint.
16. Using a plain `new Error()` in any request-path code — the error handler only formats `AppError` subclasses. Plain errors produce unformatted 500 JSON.
17. Throwing `new AppError(message, status)` with a generic HTTP-status code when a more specific code exists — clients need specific codes (`CERTIFICATE_EXPIRED`, `ISSUER_FORBIDDEN`, etc.) to react correctly without parsing `detail` strings. Always import from `src/constants/error-codes.js` and pass the code as the third argument.
18. Adding a new `AppError` throw without adding the code to `src/constants/error-codes.js` first — codes defined inline as string literals are not documented, not discoverable, and can silently diverge across call sites.
19. Adding a new notification type without updating **both** CHECK constraints (`chk_notifications_type` in migration 044 and `chk_notification_preferences_type` in migration 045) — the INSERT will fail at runtime. Also add the type to `src/constants/notification-types.js`.
20. Calling `notificationService.runChecksForTenant()` directly from a tenant request — cert checks are now API-owned and run by the admin scheduler (`POST /v1/admin/jobs/notifications`). No sync endpoint is exposed to tenants. `runCertChecksForTenant` is an exported function for the scheduler, not for controllers.
21. Forgetting to fire `webhookDeliveryService.fanOut(notification)` after creating or updating a notification — all new notification creation paths must fan out to webhook subscribers. The pattern is `if (notification) fireWebhookFanOut(notification)` using the lazy-require helper in `notification.service.js`.
22. Returning the webhook secret in PATCH/list responses — the secret is shown **once only** at registration (in the `POST /v1/webhooks` response). Never include `row.secret` in any other response shape.
23. Relying on a multer instance's `limits.fileSize` to cap one specific field when the same instance handles other fields too — the limit applies to the whole request, not per field (e.g. registration's multer covers both the P12 cert and the logo with one 10 MB ceiling). When a field needs its own stricter cap, check `req.files.<field>[0].size` manually in the controller and throw `AppError` with `INVALID_FILE_UPLOAD`.
24. Adding a new column called `updated_at` to a table without also adding a `trg_<table>_updated_at` trigger (see migration 051, `set_updated_at()`) — there is no ORM, so without the trigger nothing bumps the timestamp and it silently drifts (this happened to `issuers.updated_at` via `updateLogo`/`updateCertificate` before the trigger existed).
25. Adding `isArray()` validation for a field on a route that accepts `multipart/form-data` (any route with a file upload) without a `customSanitizer` that JSON-parses string values first — multipart fields never auto-deserialize a JSON string into an array, so a client sending a JSON-encoded array (the only way to send one over multipart) always fails `isArray()`. See the `arrayFromJsonString` helper in `src/validators/issuer.validator.js` (mirrors the one in `registration.validator.js`) for the pattern; apply it to `documentTypes`/`initialSequentials`-shaped fields on any future multipart route.
26. Computing a subscription's next `current_period_start`/`current_period_end` from `new Date()` ("now") instead of the OLD `current_period_end` — an admin reviewing a renewal/upgrade proof a few days early or late would then drift the billing date forward or backward every cycle. Always anchor to the old `current_period_end` via the shared `addBillingPeriod()` helper in `subscription.service.js`, used by `applyRenewalIfLinked` and `applyScheduledTierChanges`'s period rollover. Two legitimate exceptions where "now" IS the right anchor: `activateIfLinked` (no prior period exists — this is the first cycle) and `resetPeriodOnPromotion` (deliberately restarting the clock at the moment the tenant enters production).
27. Calling `subscriptionService.processDueRenewals()` before `applyScheduledTierChanges()` in the admin subscriptions job — a downgrade that hasn't been applied yet still has its old (soon-to-be-stale) `current_period_end`, and `processDueRenewals`'s expiry check would wrongly flag it as having run out the grace period. `applyScheduledTierChanges()` must run first in the same job tick so its period rollover lands before the expiry check reads `current_period_end`.
28. Reading `subscription.tier`/`subscription.billing_interval` directly in any code that displays a `payments` row (email templates, notifications) — for a `TIER_CHANGE` payment, the subscription still reflects its CURRENT tier/interval, not what's being purchased, until the change actually applies. Always use `payment.target_tier || subscription.tier` and `payment.target_billing_interval || subscription.billing_interval` (the fallback correctly covers `INITIAL`/`RENEWAL` payments, which never set `target_tier`). Also always read `payment.total_amount` for a displayed dollar figure, never `payment.amount` — the latter is the pre-IVA base imponible, not what the tenant actually transfers. Every payment-display surface (`payment-reviewed.js`, `payment-proof-submitted.js`, `notification.service.js`'s `createPaymentReviewed`) follows this same pattern.
29. Assuming `linkInvoice`'s `document.sandbox` branch only ever needs to handle the subscription's INITIAL activation. A sandbox self-billed invoice can just as easily fund a `TIER_CHANGE` or `RENEWAL` payment (a tenant can request a tier change, or run a subscription long enough to hit a renewal, entirely within sandbox before ever promoting). Any future code touching this path must resolve the payment's purpose the same way the production path does (`findPendingTierChangeBySubscriptionId`/`findPendingRenewalBySubscriptionId`, computed *before* the sandbox/production fork — see `linkInvoice`) and route accordingly (`linkSandboxDocument`), rather than unconditionally treating every sandbox-linked document as an activation. This was a real shipped bug: the tier flip silently no-op'd, leaving the subscription and tenant on the old tier even after the invoice was linked and authorized.
30. Adding a new authenticated route without deciding how it handles a `SUSPENDED` tenant. `authenticate` no longer blocks suspension itself — a new route in a router that applies `requireNotSuspended` per-route (currently `documents`, `subscriptions`, `tenants`, `payments`) is reachable by a suspended tenant unless you explicitly add the middleware to that route. A new router should default to `router.use(requireNotSuspended)` right after `router.use(authenticate)` unless it specifically needs a read-while-suspended exception.
31. Checking `subscriptionModel.findActiveByTenantId()` (ACTIVE only) instead of `findActiveOrPendingByTenantId()` (anything not `CANCELLED`/`EXPIRED`) before deciding whether to start a new subscription. `tenant.service.js`'s `promote()` used to do exactly this — a tenant who'd pre-subscribed but was still `PENDING_PAYMENT`/`PAYMENT_RECEIVED`/`INVOICE_PROCESSING` (payment uploaded, awaiting operator review) fell through the `ACTIVE`-only check and got a second subscription/payment kicked off on promotion, alongside the one already in flight. Any code deciding "does this tenant already have billing in progress" needs the broader in-flight check, not just `ACTIVE`.
32. Adding a file that's read from the filesystem at **runtime** (not just at build/authoring time) under a path `.dockerignore` excludes wholesale. `.dockerignore` excludes `docs` and `*.md` entirely (so documentation doesn't bloat the deploy image), but `agreement.service.js`'s `publish()` reads `docs/agreements/*.md` on every `POST /v1/admin/agreements` call — those files needed explicit `!docs/agreements` / `!docs/agreements/*.md` negation patterns or the built image silently ships without them, throwing an unhandled `ENOENT` in every deployed environment despite working locally (no Docker layer in the loop there). Any future runtime-read source file living under `docs/` needs the same carve-out.
33. Leaving Cloudflare's zone-wide Email Obfuscation enabled on a Cloudflare-proxied API hostname. `GET /v1/agreements/:type` and `GET /v1/tenants/agreements/:type` serve HTML with real embedded contact email addresses (`{{soporte.email}}`/`{{operador.email}}`, already substituted into the stored snapshot — see "Legal documents and tenant acceptance" above). Cloudflare's Email Obfuscation rewrites those addresses into `<span data-cfemail="...">` placeholders backed by a relative `/cdn-cgi/l/email-protection` decode script; when comprobify-web fetches this HTML server-side and proxies it to the browser, that relative script 404s against the frontend's own domain, so addresses render as `[email protected]`. Fixed via a Cloudflare Configuration Rule scoping Email Obfuscation → Off to just `api.comprobify.com` / `api-staging.comprobify.com` — see `docs/deployment.md`'s "Cloudflare configuration" section. Any new Cloudflare-proxied API hostname must be added to that rule's expression or its agreement pages silently break the same way.
34. Writing admin-submitted agreement edits back to `docs/agreements/*.md` (e.g. via `fs.writeFileSync`) instead of straight into `agreements.content_markdown`. Render's container filesystem is ephemeral and rebuilt from the git-committed image on every deploy (see the Releasing section) — a runtime file write is invisible to the next deploy and, if the admin API ever runs more than one instance, invisible to other instances too. `publish()`'s `contentMarkdown` parameter exists specifically so admin edits go straight to the DB and never touch the filesystem; any future "edit and republish" UI must keep using that path, not resurrect a file-write step.
35. Using `payments.invoice_document_id IS NULL` as the "not yet applied" signal for a `TIER_CHANGE`/`RENEWAL` payment. It works for production (`linkInvoice` sets it when linking), but `linkSandboxDocument` never sets it at all (see the FK/id-collision reasoning above) — so that check would keep matching an already-applied sandbox payment forever, permanently blocking further tier-change requests via the `TIER_CHANGE_ALREADY_PENDING` guard and leaving `linkInvoice` able to re-process the same stale payment. `payment.model.js`'s `findPendingTierChangeBySubscriptionId`/`findPendingRenewalBySubscriptionId`, and `subscription.model.js`'s `findDueForRenewalReminder` (its `NOT EXISTS` subquery has the identical "is a renewal already open" check), all gate on `period_start IS NULL` instead — the one column both environments stamp exactly once, at the moment the payment is actually applied. Any future query needing "has this TIER_CHANGE/RENEWAL payment been applied yet" must use the same column, not `invoice_document_id`.
36. Unconditionally setting `subscriptions.status = 'PAYMENT_RECEIVED'` in `reviewPayment()` whenever a payment is `VERIFIED`, regardless of `payment.purpose`. That transition is a step in the INITIAL activation flow only (`PENDING_PAYMENT` → ... → `PAYMENT_RECEIVED` → `INVOICE_PROCESSING` → `ACTIVE`) — a `TIER_CHANGE`/`RENEWAL` payment's subscription is already `ACTIVE` when its payment is reviewed (both are only opened for an `ACTIVE` subscription), and nothing downstream restores `ACTIVE` for the common immediate-apply tier-change path (`applyTierChangeIfLinked`'s non-deferred branch only calls `subscriptionModel.applyTierChange()`, which touches `tier`/`billing_interval`, never `status`). Demoting it here left the subscription permanently stuck at `PAYMENT_RECEIVED` even after the tier flip fully applied — which then silently broke every `findActiveByTenantId`-gated flow for that subscription (further tier-change requests, renewal reminders, expiry checks). `reviewPayment()` now only demotes to `PAYMENT_RECEIVED` when `payment.purpose === 'INITIAL'`.
37. Confusing `subscriptions.initial_invoice_document_id` (migration 072, renamed from `invoice_document_id`) with `payments.invoice_document_id` — they are two different FKs on two different tables with two different lifetimes. `subscriptions.initial_invoice_document_id` is write-once: set only by `linkInvoice`'s INITIAL branch and read only by `activateIfLinked` (via `subscriptionModel.findByInitialInvoiceDocumentId`) as its dispatch key for "which document just finished this subscription's first activation." Every subsequent funding event (`TIER_CHANGE`, `RENEWAL`) writes to its own `payments` row's `invoice_document_id` instead (read by `applyTierChangeIfLinked`/`applyRenewalIfLinked` via `paymentModel.findByInvoiceDocumentId`) — never to the subscription's column. Do not add code that repoints `subscriptions.initial_invoice_document_id` to a later invoice; that would collide with `activateIfLinked`'s dispatch key and erase the record of what actually started the subscription.
38. Calling `tenantModel.updateTier(id, tier)` without a matching `tenantQuotaService.setCap(id, tier)` call right after it (migration 073 split what used to be one `UPDATE tenants SET subscription_tier, document_quota` statement into two independent calls, one per table, since quota moved to `tenant_quotas`). `tenantModel.updateTier` only ever touches `subscription_tier` now — every one of its ~10 call sites (`admin.service.js`'s `updateTenantTier`, and 9 in `subscription.service.js`) must pair it with `tenantQuotaService.setCap` or the tenant's tier changes while their quota cap silently stays at the old tier's value.
39. Adding or changing a `documents.status` transition in `src/constants/document-state-machine.js` without also updating the `enforce_document_state_transition()` DB trigger function (migration 027, `CREATE OR REPLACE FUNCTION`, reused by both `public.documents` and `sandbox.documents` via `trg_..._state_transition`). This trigger hardcodes the identical transition graph **independently** of the JS state machine — it is not derived from it. Migration 074 (adding `PENDING_SEND`) had to update both in the same migration; updating only the JS side would pass `assertTransition()`'s check in application code but still get rejected by Postgres with `Invalid document state transition: X -> Y` at the `UPDATE` itself.
