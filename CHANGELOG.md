## [Unreleased]

### Changed
- **A `SUSPENDED` tenant can now still view — but not use — the Service.** Previously `authenticate` middleware rejected every request outright for a suspended tenant, reads included. The suspension check moved to a new `src/middleware/require-not-suspended.js`, applied selectively: a suspended tenant can still list/download their own existing documents (RIDE, XML), view their subscription and payment-proof history, and see their account status/agreements/event log — all read-only, nothing that "uses" the Service further. Every write stays blocked, plus one read that isn't purely passive: `GET /:accessKey/authorize` makes a live SRI call and can fire the authorization email, so it's excluded from the exception. Routers with no exception (`issuers`, `api-keys`, `catalogs`, `notifications`, `webhook-endpoints`) keep the same blanket block as before, just moved one middleware over.

### Added
- **`GET /v1/documents/:accessKey/sri-responses`** — returns the raw SRI reception/authorization call outcomes recorded for a document (`operationType`, `status`, `messages`, `createdAt`, newest first), omitting the internal raw SOAP body. Reuses the same `authenticate → resolveIssuer` chain as every other document read endpoint. `sriResponseModel.findByDocumentId()` previously only queried `public.sri_responses` regardless of environment — it now takes a `sandbox` flag and selects the right schema table, mirroring `create()`'s existing pattern.
- **`PATCH /v1/admin/tenants/:id/status` now logs a `STATUS_CHANGED` tenant event** (`{ from, to }`, migration 070) — suspending or reactivating a tenant was previously silent, unlike the tier-override endpoint's `TIER_CHANGED` event. Shows up in both `GET /v1/tenants/events` and `GET /v1/admin/tenants/:id/events`.
- **Separate sender address for invoice emails + staging banner + do-not-reply notices** — new optional `EMAIL_FROM_DOCUMENTS` env var lets invoice/document emails use a different sender address than every other transactional email (e.g. `facturas@` vs. `notificaciones@`); falls back to `EMAIL_FROM` when unset, so existing deployments need no change. When `APP_ENV=staging`, every outgoing email now carries a bold, localized "TEST ENVIRONMENT" / "ENTORNO DE PRUEBAS" banner in both `text` and `html` (`email.service.js`'s new `applyStagingBanner()`, applied uniformly across all 6 email types) so a test message can never be mistaken for a real one. The verification email's disclaimer now also explicitly says "do not reply" in both languages, matching the other 5 templates which already had it.
- **Multi-file, historical payment proof uploads** — `PATCH /v1/payments/:id/proof` now accepts up to 5 files per request and never overwrites anything already uploaded (including across a rejection-and-resubmission cycle) — proof files moved from three columns on `payments` (`proof_file`/`proof_filename`/`proof_mime_type`) into their own append-only `payment_proofs` table (migration 069), one row per file, mirroring the `document_events`/`tenant_events` pattern. New tenant-facing `GET /v1/payments/:id/proofs` (list, active-only), `GET /v1/payments/:id/proofs/:proofId` (download one), and `DELETE /v1/payments/:id/proofs/:proofId` (soft delete — `active = false`, never a hard delete, per CLAUDE.md rule #7); a tenant-deleted file drops out of their own list/download but stays fully visible to the operator. New admin-facing `GET /v1/admin/payments/:id/proofs` (list, includes inactive) and `GET /v1/admin/payments/:id/proofs/:proofId` (download any, active or not) replace the old single-file `GET /v1/admin/payments/:id/proof`. A new cumulative cap (`MAX_ACTIVE_PROOFS_PER_PAYMENT = 10`, across every upload attempt) prevents unbounded growth from repeated resubmission — new error code `PROOF_FILE_LIMIT_REACHED`. New `src/models/payment-proof.model.js`. **Breaking:** the old `GET /v1/payments/:id/proof` and `GET /v1/admin/payments/:id/proof` single-file endpoints are gone; `payments` rows no longer carry `proof_file`/`proof_filename`/`proof_mime_type` in any response.
- **Predefined payment-rejection reason codes** — `PATCH /v1/admin/payments/:id/review`'s `rejectionReasonCode` (required when rejecting) replaces the old free-text `rejectionReason` entirely: one of `AMOUNT_MISMATCH`, `TRANSFER_NOT_FOUND`, `WRONG_ACCOUNT`, `ILLEGIBLE_PROOF`, `DUPLICATE_SUBMISSION`, `OTHER` (`src/constants/rejection-reasons.js`), validator-whitelisted and re-checked at the service layer (400 `INVALID_REJECTION_REASON` otherwise). Stored as `payments.rejection_reason_code` (migration 068, drops the old `rejection_reason` column). Lets a frontend map the stable code to its own localized message instead of displaying whatever an admin typed — the tenant-facing `rejection_reason_code` returned from `GET /v1/subscriptions/me` stays code-only for the same reason, same philosophy as any other `error.code`. The rejection email (`payment-reviewed.js`) and the `PAYMENT_REJECTED` notification each independently resolve the code to a human message (email: localized via each locale's `email.paymentRejected.rejectionReasonLabels`; notification: hardcoded English, matching the rest of `notification.service.js`'s un-localized messages) — the notification's `metadata.rejectionReasonCode` still carries the raw code too, for consumers doing their own mapping.
- **Billing-interval changes on `POST /v1/subscriptions/change-tier`** — the endpoint now accepts an optional `billingInterval` (`MONTHLY`/`YEARLY`) alongside `tier`, so a tenant can switch cadence (e.g. monthly STARTER → yearly GROWTH) in the same call, not just tier. Same-interval upgrades/downgrades behave exactly as before. Any change that also switches billing interval — whichever direction the tier moves — is always deferred to `current_period_end` and billed at the new tier+interval's full sticker price, never prorated across mismatched cadences: the current period runs out as already paid for, and the new cadence starts its own fresh, fully-paid period. New columns `subscriptions.pending_billing_interval` / `payments.target_billing_interval` (migration 066); `applyTierChangeIfLinked` now checks `target_billing_interval` to decide immediate-apply vs. defer-to-period-end, and `applyScheduledTierChanges`'s period rollover now uses the *new* interval rather than always the old one. New error code: `INVALID_BILLING_INTERVAL`. `TIER_CHANGE_REQUESTED`/`TIER_CHANGE_SCHEDULED`/`TIER_CHANGED` tenant events now carry `fromBillingInterval`/`toBillingInterval` in `detail`.
- **`GET /v1/tenants/events`** and **`GET /v1/admin/tenants/:id/events`** — the `tenant_events` audit trail (written since early on by registration, subscription, payment, and tier-change code) is now readable back, tenant-facing and admin-facing respectively. This is the only place a tenant can see the full sequence of subscription/tier changes over time, not just current state. Shared response shape via new `src/presenters/tenant-event.presenter.js`.
- **IVA-inclusive pricing + per-payment IVA audit trail** — subscription tier prices updated to IVA-inclusive all-in amounts (STARTER $20, GROWTH $90, BUSINESS $230 monthly; yearly ×10). `IVA_RATE = 0.15` exported from `subscription-tiers.js` as the single source of truth. Every new payment row now stores `amount` (base imponible), `iva_rate`, `iva_amount`, and `total_amount` so the full tax breakdown at time of payment creation is permanently auditable. `GET /v1/tiers` now exposes `ivaRate`, `priceMonthlyUsdBase`, `priceMonthlyUsdIva`, `priceYearlyUsdBase`, and `priceYearlyUsdIva` alongside the existing all-in prices. Price change clause added to Terms of Service (30-day notice; monthly plans: next cycle; annual plans: next annual renewal).
- **`GET /v1/payments/:id/proofs/:proofId`** — tenants can now download a specific proof-of-transfer file they previously uploaded for a payment. Ownership enforced via the subscription join — a tenant can only retrieve their own files.
- **`DELETE /v1/subscriptions`** — tenants can now schedule a cancellation at the end of the current billing period. No refund is issued; the subscription keeps running at the current tier until `current_period_end`, then the tenant drops to FREE and the subscription is closed (`CANCELLED`). Setting `pending_tier = 'FREE'` reuses the existing scheduled-downgrade infrastructure (`applyScheduledTierChanges`, daily admin job) — no new job needed. New error code: `CANCELLATION_ALREADY_PENDING`.
- **Recurring subscription renewals** — `subscriptions.current_period_end` is now a real recurring billing date: `subscriptionService.processDueRenewals()` (called by `POST /v1/admin/jobs/subscriptions`, after `applyScheduledTierChanges()`) opens a `RENEWAL` payment + reminder ~7 days before it's due, and downgrades an unpaid subscription to FREE ~7 days after if no renewal ever completes. Renewal payments ride the exact same proof-upload/admin-review/link-invoice pipeline as the initial subscription. Period math (`addBillingPeriod`) is now anchored to the old period end everywhere a period extends, never to "now," so an early or late admin review can't drift the billing date; a downgrade also rolls its period forward for free so it re-enters the renewal cycle at its new tier instead of being mistaken for an expired one.
- **Payment review and renewal notifications/emails** — `reviewPayment()`'s `VERIFIED`/`REJECTED` decision now fires a tenant-facing notification (new `PAYMENT_VERIFIED`/`PAYMENT_REJECTED` types, fanned out to webhooks) and email, covering every payment purpose (initial subscription, tier change, renewal) uniformly. The renewal reminder and the eventual expiry-to-FREE each fire their own notification (`SUBSCRIPTION_RENEWAL_DUE`, `SUBSCRIPTION_EXPIRED`) and email — a tenant should never be silently downgraded. Separately, `submitPaymentProof()` now emails the operator (`ADMIN_NOTIFICATION_EMAIL`, optional — skipped if unset) so there's a signal that a proof is waiting for review, instead of relying on the operator to remember to check.
- **`POST /v1/subscriptions`** — lets a tenant start a paid subscription on its own, independent of promotion. Works while still in sandbox (so a tenant can pay before going to production) or any time after, requiring only a verified email (same gate `POST /v1/tenants/promote` already used). `POST /v1/tenants/promote` now checks for an already-`ACTIVE` subscription before honoring its own optional `tier`/`billingInterval` fields — if one exists (e.g. started via this new endpoint), tier selection is skipped entirely and the response surfaces the existing subscription instead of starting a new one.
- **Subscription + payment pipeline** — new `subscriptions`/`payments` tables track a tenant's paid-tier billing while no payment gateway exists. A subscription only reaches `ACTIVE` once its linked invoice document is SRI-`AUTHORIZED`, never merely on payment. Tier selection happens at `POST /v1/tenants/promote` (new optional `tier`/`billingInterval` fields) — promotion itself never waits on payment. The tenant uploads proof of the SPI transfer themselves via `PATCH /v1/payments/:id/proof` (new, multipart, their own API key); the operator reviews it (`GET /v1/admin/payments/:id/proofs` / `.../proofs/:proofId`) and decides with one endpoint, `PATCH /v1/admin/payments/:id/review` (`{ decision: "VERIFIED" | "REJECTED", rejectionReasonCode }` — a predefined reason code required when rejecting), then links the self-billed invoice (`PATCH /v1/admin/subscriptions/:id/link-invoice`). A rejected payment can be re-submitted with new proof (only `VERIFIED` blocks further uploads). `document-transmission.service.js` fires `subscriptionService.activateIfLinked()` fire-and-forget on every document authorization (no-op unless that document is a subscription's linked invoice).
- **`GET /v1/subscriptions/me`** — tenant-facing, returns the full subscription/payment history (with `rejection_reason_code` when applicable) since there's no notification when a review or activation completes.
- **Yearly billing** — `priceYearlyUsd` (2 months free vs. monthly) added for Starter/Growth/Business. New public `GET /v1/tiers` (no auth) exposes the full tier catalog — quota, both prices, overage rate, limits — for a pricing page.
- **`PATCH /v1/admin/tenants/:id/tier`** (the direct admin tier override) now logs a `TIER_CHANGED` tenant event — previously silent.
- **Document type tier gating** — `POST /v1/issuers/:id/document-types` and the `documentTypes` field on `POST /v1/issuers` now reject activating a type not included in the tenant's plan (402 `DOCUMENT_TYPE_NOT_IN_TIER`). Free/Starter are limited to facturas (`01`); Growth/Business also get credit notes (`04`). Downgrading is grandfathered — already-active types on existing issuers keep working; only activating *new* gated types is blocked.
- **`POST /v1/subscriptions/change-tier`** — lets a tenant with an `ACTIVE` subscription upgrade or downgrade tier. No payment gateway exists, so this rides the same manual proof/review pipeline as the initial subscription. Upgrades take effect immediately, gated on a prorated payment (the price difference scaled by the remaining time in the current billing period) — uploaded via the same `PATCH /v1/payments/:id/proof`, reviewed and invoice-linked the same way, applied on invoice authorization (or immediately, with no payment step, if the prorated amount rounds to $0). Downgrades are scheduled with no payment owed — the current period is already paid for at the higher tier — and applied at `current_period_end` by a new admin job, `POST /v1/admin/jobs/subscriptions` (same external-cron pattern as the notification job, daily cadence is enough). New error codes: `NO_ACTIVE_SUBSCRIPTION`, `TIER_CHANGE_NO_OP`, `TIER_CHANGE_ALREADY_PENDING`.

### Fixed
- **`POST /v1/admin/tenants`** (Create Tenant) now rejects an unrecognized `subscriptionTier` (400 `INVALID_TIER`) instead of silently storing it and falling back only on document quota — matches the validation `PATCH /v1/admin/tenants/:id/tier` already had.
- **`POST /v1/admin/agreements`** (Publish Agreement) now throws a proper `AppError` (500, new code `OPERATOR_CONFIG_MISSING`) when `OPERATOR_NAME`/`OPERATOR_RUC`/`OPERATOR_EMAIL` aren't configured, instead of a plain `Error` that the central error handler couldn't format into RFC 7807 shape.
- **Certificate parsing** (`certificate.service.js`) no longer crashes with an unhandled `TypeError` on a P12 file containing only a single certificate (no CA chain) — now rejects cleanly with `400 CERTIFICATE_INVALID`.
- **Document audit trail** — an authorization email that's intentionally skipped (e.g. no `buyer_email` on file) now logs a new `EMAIL_SKIPPED` event instead of being indistinguishable from a genuine `EMAIL_FAILED` send failure. `documents.email_status` already had a `SKIPPED` value (migration 029); `document_events.event_type` did not until now (migration 067).
- **`subscription.service.test.js`** — fixed a latent test-isolation bug where `linkInvoice`'s tests didn't reset two payment-lookup mocks in their own `beforeEach`, letting state from an unrelated earlier test leak in under randomized execution order.
- **Payment-review emails and notifications showed the wrong plan for a `TIER_CHANGE` payment** — `payment-reviewed.js` (tenant email), `payment-proof-submitted.js` (operator email), and `notification.service.js`'s `createPaymentReviewed()` all read `subscription.tier`/`subscription.billing_interval` directly, which is the subscription's *current* plan, not the one being purchased — a monthly-STARTER-to-yearly-GROWTH change showed "STARTER" everywhere instead of "GROWTH." Fixed to `payment.target_tier || subscription.tier` / `payment.target_billing_interval || subscription.billing_interval` (the fallback correctly covers `INITIAL`/`RENEWAL` payments, which never set `target_tier`) — see CLAUDE.md Common Mistakes #28. Also fixed the displayed amount, which read `payment.amount` (the pre-IVA base) instead of `payment.total_amount` (what's actually transferred); billing frequency is now shown alongside tier in both emails. The tenant-facing rejection email's next-steps text also no longer references the raw `PATCH /v1/payments/:id/proof` endpoint path.
- **A sandbox-linked `TIER_CHANGE`/`RENEWAL` invoice silently left the subscription and tenant on the old tier.** `linkInvoice`'s `document.sandbox` branch unconditionally treated every sandbox-linked document as the subscription's INITIAL activation, running *before* the purpose-detection logic below it — so a sandbox tier-change payment's invoice would get linked and authorized, but the subscription/tenant tier never actually flipped (and even if it had, it would've used `subscription.tier`, the *old* tier, not the target one). Fixed by resolving the payment's purpose (`TIER_CHANGE`/`RENEWAL`/`INITIAL`) *before* the sandbox/production fork and routing accordingly through a new `linkSandboxDocument` helper — see CLAUDE.md Common Mistakes #29. **Also:** while `tenant.sandbox === true`, `requestTierChange`/`linkInvoice` no longer prorate or defer a tier/interval change at all — every sandbox change is now priced at the target plan's **full sticker price** and applies **immediately** once its invoice authorizes, since a sandbox `current_period_end` is discarded entirely at promotion (`resetPeriodOnPromotion`) and has nothing meaningful to prorate against or defer to. New `requestSandboxTierChange` function.

### Changed
- **Privacy Policy no longer claims per-request API logging that doesn't exist** — §2's data table and §6 "Registros" both described logging IP, timestamp, and endpoint for every single API request, plus an "identificador de cuenta" per request. No such system exists: there's no request-logging middleware (no morgan/winston/pino, no access-log table), and Sentry is explicitly configured with `sendDefaultPii: false`, so it doesn't capture IPs either. The only IP/user-agent actually stored anywhere in the schema is `tenant_agreements.ip`/`.user_agent` (migration 061), recorded once, specifically as acceptance evidence when a tenant accepts TERMS/PRIVACY/DPA. Both sections now describe only that. Also removed a dead `acceptanceContext` parameter from `registrationService.register()` — `registration.controller.js` captured `req.ip`/user-agent into it, but it was never used inside the function, so registration itself never stored an IP despite appearing to.
- **Privacy Policy and Terms of Service updated: individual users, comprobify-web data, clearer price-change mechanics, and a support-routed contact address.** "El Cliente" in the Privacy Policy no longer says "la empresa que usa Comprobify" — individuals (persona natural) use the Service too, matching wording the Terms already had. Both documents now describe comprobify-web's additional data storage (buyer catalog, product/service catalog) the same way the DPA does: conditional on using the web interface, worded openly enough to cover further data categories added as the system evolves, and with the Privacy Policy explicit that catalog entries aren't tax records and so stay deletable on request unlike data in an authorized comprobante. The Privacy Policy's subencargados table gets the same Vercel/Neon additions as the DPA's. The Terms' price-change clause is rewritten to state the actual rule explicitly (a renewal that falls within the 30-day notice window still bills at the old price; the new price only takes effect at the first renewal on or after the 30 days elapse) instead of two cycle-specific sentences that left the mid-cycle case ambiguous. "La dirección registrada en la cuenta" is now "la dirección de correo electrónico de la cuenta del Cliente" — `tenants.email`, the address every subscription/billing email in this system actually uses; `issuers` has no email column, so a per-issuer framing wasn't accurate. New `{{soporte.email}}` template token (`agreement.service.js`'s `publish()`, resolved from `ADMIN_NOTIFICATION_EMAIL` with a fallback to `OPERATOR_EMAIL` so a published document never bakes in a blank address) replaces `{{operador.email}}` everywhere a document invites the reader to write in (account termination, "para consultas," exercising data rights) — `{{operador.email}}` stays reserved for identifying the legally responsible party, now used only in the DPA's opening clause.
- **Removed redundant date lines from all three legal documents** — Terms of Service and Privacy Policy each had a "Última actualización: {{fechaVersion}}" line right below the title, on top of the "Versión: X" already shown in the disclaimer box above it; the DPA's opening sentence had its own "{{fechaDocumento}}" execution date. All three now rely solely on the disclaimer's "Versión: X" as the one recency indicator per document, matching how the DPA already worked. Also fixes a latent bug: the public `GET /v1/agreements/:type` route never substituted `{{fechaVersion}}` (it resolved a differently-named `fecha` key), so that line rendered as a literal, unreplaced token on that route — moot now that the line is gone. Removed the now-dead `fechaVersion`/`fechaDocumento` fields from `tenant-agreement.service.js`'s `buildValues()`.
- **DPA content updated to cover the comprobify-web buyer catalog** — the DPA previously stated the Service stores no buyer data outside authorized comprobantes, which stopped being accurate once the web frontend gained a reusable buyer-catalog feature. §3 and §7 now describe this conditionally ("only if the Responsable uses comprobify-web"), and explicitly carve buyer-catalog entries out of the SRI 5/7-year non-deletion rule (they're not tax records, so they stay deletable on request, unlike data already baked into an authorized comprobante). §6's subencargados table adds Vercel (web hosting) and broadens Neon's entry to cover the web app's separate database, both flagged as applying only to web-interface users. Product/service catalogs stay out of the DPA's scope entirely — they're the Responsable's own business data, not a third party's personal data. Content-only change to `docs/legal/data-processing-agreement.md`; publishing a new version via `POST /v1/admin/agreements` is still a separate, manual step.
- **Legal documents now render as a styled, formal-looking page instead of raw unstyled HTML** — `GET /v1/agreements/:type` and `GET /v1/tenants/agreements/:type` now wrap the disclaimer + rendered markdown in a complete HTML page (`agreementService.wrapDocumentHtml()`): serif typography, justified body text, a centered "paper" container, styled heading hierarchy, and a `<title>` tag (e.g. "Términos de Servicio — Comprobify") for the browser tab. No content or endpoint contract changes — same response shape, just formatted.
- **Legal document disclaimer now points to the support inbox, not the operator's personal email** — `GET /v1/agreements/:type` and `GET /v1/tenants/agreements/:type`'s prepended "questions before accepting?" notice now uses `ADMIN_NOTIFICATION_EMAIL` instead of `OPERATOR_EMAIL`. `OPERATOR_EMAIL` still appears elsewhere as the legally responsible party's contact (baked into the template content itself via `{{operador.email}}` — data-controller/termination-request contact), unaffected by this change.
- **FREE tier document quota reduced from 100 to 5 documents/month** — the prior limit let a real freelancer or small business run indefinitely without ever needing to upgrade.
- **Document quota no longer counts sandbox documents** — `tenants.document_count` only increments for production document creation, so testing against SRI pruebas never consumes a tenant's plan quota.
- **Starter/Growth/Business document quotas cut to 200/1,000/4,000 (from 1,000/5,000/20,000)** — the old figures granted far more volume than any published competitor tier in the Ecuadorian/regional market (Datil's cheapest tier caps at 12/month; NubeFact's flagship plan caps at 500/month). Overage rates raised in tandem (Starter $0.04→$0.30, Growth $0.025→$0.15, Business $0.015→$0.08/invoice).
- **`priceMonthlyUsd` / `overagePerDocumentUsd` added to `src/constants/subscription-tiers.js`** — pricing now has one source of truth alongside the limits it's priced against, instead of living only in a planning doc. Not billed or enforced anywhere yet — no payment gateway is wired up (see `NEXT_STEPS.md` #9).
- **Removed `STRATEGY.md`** — its pricing model is superseded by the constants above; its competitive analysis, payments-in-Ecuador findings, and roadmap had served their purpose for the planning phase now that the project is moving into execution.

## [0.6.0] — 2026-06-27

### Added
- **`PATCH /v1/issuers/:id/activate`** — reactivates a soft-deleted issuer. Re-runs the same branch/issue-point plan-limit checks as `POST /v1/issuers` (`BRANCH_LIMIT_REACHED` / `ISSUE_POINT_LIMIT_REACHED`, 402) so deactivate-then-reactivate can't be used to exceed a tenant's plan caps.
- **`PATCH /v1/issuers/:id`** — edit `tradeName` and/or `branchAddress` for an existing issuer (at least one required). Does not accept `businessName`/`mainAddress`/`ruc` — those stay permanently tied to the RUC registration.
- **`DELETE /v1/issuers/:id`** — soft-deletes an issuer (`active = false`). Rejects with `LAST_ISSUER_CANNOT_BE_REMOVED` (400) if it's the tenant's only remaining issuer, or `ISSUER_HAS_DOCUMENTS` (400) if it has ever issued a document (checked in both the `public` and `sandbox` schemas).
- **`GET /v1/issuers/:id/sequentials`** — returns the current and next sequential number per active document type, broken out by `sandbox`/`production` environment, so the UI can show what a manual edit would set without requiring callers to do the `current + 1` math.
- **`PATCH /v1/issuers/:id/sequentials/:documentType`** — manually sets the next sequential number for one document type in one environment (`{ environment, nextSequential }`). Locks the counter row with `SELECT ... FOR UPDATE` inside the same transaction as the write, so it can't race against a concurrent `POST /v1/documents` and produce a duplicate sequential. Rejects with `SEQUENTIAL_CANNOT_DECREASE` (400) if `nextSequential` doesn't exceed the current value.
- **`GET /v1/documents/:accessKey/credit-notes`** — returns the sum of all `AUTHORIZED` credit notes already issued against a document plus the remaining balance, so callers can guard against over-crediting an invoice. Reconstructs the document's own `NNN-NNN-NNNNNNNNN` number and matches credit notes by `request_payload->'originalDocument'` (there's no FK linking them). Document-type-agnostic on the original side — doesn't assume `01`.
- **Nota de Crédito (document type `04`)** — `POST /v1/documents` now accepts credit notes referencing an original invoice (`originalDocument.documentType`/`number`/`issueDate`) plus a `motivo`. New `CreditNoteBuilder`, dedicated validator (no `payments` block; see `src/middleware/select-document-validator.js` for per-type validator dispatch), schema-aware XSD validation (`assets/nota_credito_V1.1.0.xsd`), and document-type-aware RIDE PDF (header label + a "Documento que se modifica" section in place of "Forma de pago") and authorization email subject/labels.
- **`PATCH /v1/issuers/:id/certificate`** — renews the P12 certificate (private key + X.509 cert) stored for an existing issuer, e.g. when it has expired. Reuses `certificateService.parseCertificate` + `cryptoService.encrypt` from issuer creation; updates only the targeted issuer row (sibling branches that inherited the cert via `sourceIssuerId` are unaffected). Does not touch already-signed documents — each signed invoice embeds its own certificate copy inside the XML signature. Admin override at `PATCH /v1/admin/issuers/:id/certificate` (no tenant ownership check).
- **`GET /v1/documents/stats`** — per-document-type breakdown (`issued` count + `authorizedTotal`) for the current calendar month, plus an all-time `needsAttention` count (`RETURNED` + `NOT_AUTHORIZED`). Powers the comprobify-web dashboard's revenue summary.
- **`GET /v1/tenants/me`** — resolves the tenant owning the authenticated API key (`id`, `email`, `subscriptionTier`, `status`, `documentCount`, `documentQuota`, `sandbox`). No DB lookup — echoes what `authenticate` middleware already resolved from the key. Lets a third-party app that only holds an API key (no RUC/P12 on hand) discover its numeric `tenant.id`, e.g. to link an existing account or correlate webhook deliveries.

### Changed
- **`GET /v1/documents` supports sorting and two new filters** — `sortBy` (`sequential`, `buyerName`, `issueDate`, `status`) + `sortDir` (`asc`/`desc`, defaults to `desc`) sort the list via a column whitelist; omitting `sortBy` keeps the existing `created_at DESC` default, so no behavior change for existing callers. New `buyerName` query param filters with a case-insensitive contains match (`ILIKE`). New `sequential` query param filters with a case-insensitive contains match against the zero-padded 9-digit value (`LPAD(sequential::text, 9, '0') ILIKE ...`) — `sequential` is an `INTEGER` column, so `ILIKE` can't apply directly, and the padding also matches the 9-digit format already shown in API responses. Both combine with `AND` like the existing filters. Also fixed `documentType` validation, which only accepted `01` — it now accepts all supported codes (`01`, `03`, `04`, `05`, `06`, `07`).

### Fixed
- **`POST /v1/issuers` rejected valid `documentTypes`/`initialSequentials` sent as `multipart/form-data`** — the request is multipart (it also carries the optional P12 `cert` file), and multipart fields never auto-deserialize a JSON string into an array, so a JSON-encoded array string for either field always failed `isArray()` validation. `registration.validator.js` already solved this for the identical fields on `POST /v1/register` via a `customSanitizer` that JSON-parses string values before the array check; `issuer.validator.js`'s `createBranch` now applies the same sanitizer.
- **Email retry could resend a `DELIVERED` or `COMPLAINED` email without `?force=true`** — the guard in `retrySingleEmail` only checked for `email_status === SENT`. Now blocks retry on any terminal success status (`SENT`, `DELIVERED`, `COMPLAINED`) unless `force=true` is explicitly passed.
- **Logo upload at registration could exceed 500 KB** — `POST /v1/register`'s multer instance enforces a single 10 MB whole-request limit (shared with the P12 cert field), so it could not cap the logo field individually. `registration.controller.js` now checks the logo file's size directly and throws `INVALID_FILE_UPLOAD` (400) if it exceeds 500 KB, matching the limit already enforced by `PATCH /v1/issuers/:id/logo`.

### Added
- **Issuer logo on RIDE PDFs** — issuers can now upload a company logo that renders in the top-left corner of every RIDE PDF (including email attachments). Logo is stored as `BYTEA` in `issuers.logo` (migration 050, replaces the unused `logo_path` column). Two ways to set it:

### Fixed
- **RIDE PDF "Forma de pago" section missing bottom border** — the last payment row had no closing line. The outer `strokeBox` only provided the overall section border; individual rows drew an `hline` at their top but not their bottom. A closing `hline` is now drawn after the last payment row.
- **Rebuild silently dropped a corrected `buyer_email` and left stale line items** — `buyer_email` was missing from `MUTABLE_EXTRA_COLUMNS` in `document.model.js` and was never passed to `updateStatus`, so a corrected buyer email on `POST /:key/rebuild` was ignored. Separately, line items (including tax `rateCode` changes) were never deleted and re-inserted on rebuild, leaving the original rows in `document_line_items` unchanged alongside the new signed XML. Both fixes are now wrapped in a single transaction (`document-rebuild.service.js`) so the document update, line-item replacement (`documentLineItemModel.deleteByDocumentId` + `bulkCreate`), and `REBUILT` audit event are atomic.
- **Bypass-mode writes on tenant-scoped tables could violate RLS if the issuer context was omitted** — `documentModel.updateStatus`, `documentLineItemModel.bulkCreate`, and `documentEventModel.create` now explicitly qualify the `sandbox`/`public` schema prefix in their `db.query()` (no client, no issuerId) bypass path, so a caller that omits the issuer context can no longer produce an RLS violation.
- **Mailgun webhook events for sandbox documents never updated `email_status`** — `findByEmailMessageId` queried `public.documents` only, so sandbox documents (created while `tenant.sandbox = true`) never transitioned from `SENT` to `DELIVERED`/`FAILED`/`COMPLAINED`. The query now `UNION ALL`s across `public.documents` and `sandbox.documents`, returns a `sandbox` boolean on the result row, and threads it through to both `updateEmailStatus` and `documentEventModel.create` so the update and audit event land in the correct schema. See ADR-010.
- **Mailgun webhook audit events violated RLS** — `documentEventModel.create` calls in the webhook path passed `issuerId = null`, routing through the no-context `db.query()` bypass, which the `document_events` RLS policy rejects for `INSERT`. `document.issuer_id` (available from the `findByEmailMessageId` result) is now passed through, routing the insert via `db.queryAsIssuer` so the RLS context is set correctly.

## [3.0.0] — 2026-03-01

### Breaking Changes

- **`documentType` is now required** on `POST /api/documents` — no silent default. Callers that previously omitted this field and relied on the implicit `'01'` default will receive a `400` validation error.
- **API routes renamed** from `/api/invoices/*` to `/api/documents/*`. All client integrations must update their base path.
- **`cert_path` and `cert_password_enc` columns removed** from `issuers` (migration 028). The database schema must be migrated before upgrading. These are replaced by `encrypted_private_key`, `certificate_pem`, `cert_fingerprint`, and `cert_expiry`.

### Added

- **Admin API** — `POST /api/admin/issuers`, `GET /api/admin/issuers`, `POST /api/admin/issuers/:id/api-keys`, `DELETE /api/admin/api-keys/:id` protected by `ADMIN_SECRET` (constant-time comparison). Replaces the dev seeder for issuer provisioning.

- **Sequential counter seeding** — `POST /api/admin/issuers` accepts optional `initialSequentials` (array of `{ documentType, sequential }`) to pre-seed counters for one or more document types, enabling migrating issuers that have already issued documents outside this system.
- **Multi-tenancy via Bearer API key authentication** — each request is authenticated by `Authorization: Bearer <token>`; the token resolves to an issuer row attached as `req.issuer`. Replaces the single-tenant `issuerModel.findFirst()` pattern.
- **Document state machine** — `src/constants/document-state-machine.js` defines the allowed transition graph; `assertTransition(from, to)` is called at the top of each service operation. Enforced at the DB level by `trg_document_state_transition` (migration 027) as defence in depth.
- **Document immutability triggers** — `trg_document_immutability` (migration 026) protects permanently immutable columns (`access_key`, `sequential`, `issuer_id`) and set-once authorization fields at the PostgreSQL level.
- **Email delivery** — when a document becomes `AUTHORIZED`, `emailService.sendInvoiceAuthorized()` is called fire-and-forget. Sends RIDE PDF + signed XML as attachments via Mailgun. Per-document `email_status` tracked (`PENDING` → `SENT` / `FAILED` / `SKIPPED`).
- **Email retry** — `POST /api/documents/email-retry` (batch, up to 100 docs) and `POST /api/documents/:key/email-retry` (single, `?force=true` to resend an already-sent email).
- **Idempotency key** — `POST /api/documents` accepts an optional `Idempotency-Key` header. Duplicate key + matching body → 200 replay. Duplicate key + different body → 409. Concurrent races handled via `23505` catch-and-fetch.
- **Audit trail endpoint** — `GET /api/documents/:key/events` returns the full `document_events` history for a document.
- **Rebuild** — `POST /api/documents/:key/rebuild` re-signs a `RETURNED` or `NOT_AUTHORIZED` document with corrected content, reusing the same access key and sequential.
- `ConflictError` (409) error class added to the hierarchy.
- `multer` dependency (memory storage, P12 never written to disk).
- Migrations 019–028.

### Changed

- `helpers/signer.js` signature changed from `sign(certPath, password, xml)` to `sign(privateKeyPem, certPem, xml)` — no longer reads any file from disk.
- `signing.service.js` now decrypts `issuer.encrypted_private_key` (private key PEM) instead of `issuer.cert_password_enc` (cert password).
- `document.service.js` split into five focused services: `document-creation`, `document-transmission`, `document-rebuild`, `document-email`, `document-query`.
- `invoice_details` table renamed to `document_line_items`.
- Master data tables (`clients`, `products`) removed — buyer information is stored directly on `documents`.
- `issuers` unique constraint changed from `(ruc)` to `(ruc, branch_code, issue_point_code)`.
- `document-rebuild.service.js` now reads `document.document_type` from the stored record instead of hardcoding `'01'`.

### Removed

- `cert_path`, `cert_password_enc` columns from `issuers`.
- `db/seeders/dev-issuer.js` and `seed:dev` npm script.
- `issuerModel.findFirst()` — issuers are resolved exclusively via `apiKeyModel.findByKeyHash()` during authentication.
