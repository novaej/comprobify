# ADR-005: Email Delivery via Mailgun

## Status
Accepted

## Date
2026-03-01

## Context

When an invoice becomes `AUTHORIZED` by SRI, the buyer should receive the RIDE PDF and the authorization XML automatically. The requirements were:

- Attach both files to the email (generated on the fly — no file storage)
- Spanish email body personalized with invoice details
- Track delivery status per document so failed sends can be retried
- Easy to swap the email provider in the future

Two approaches were evaluated: a **direct SMTP/nodemailer integration** and a **Mailgun HTTP API via `mailgun.js`**.

## Decision

Use **Mailgun** as the email provider, accessed via the official `mailgun.js` SDK, with a thin provider abstraction layer in `src/services/email/providers/`. The factory in `src/services/email/index.js` resolves the provider at runtime from `EMAIL_PROVIDER` env var, making swaps a one-file addition.

## Consequences

### Positive
- No SMTP server to manage or keep credentials for — Mailgun handles deliverability, SPF/DKIM signing, bounce handling, and logs.
- Mailgun's event log (delivered, opened, rejected, bounced) is queryable independently of the application, providing visibility without extra instrumentation.
- Sandbox domain allows testing email flows without sending to real inboxes.
- Provider abstraction means switching to SendGrid, SES, or another Mailgun-compatible API requires only a new file in `providers/` and an env var change.

### Negative
- External dependency — if Mailgun is unreachable the send fails (mitigated by retry tracking).
- Sandbox domain restricts delivery to pre-authorized recipient addresses during development.
- `mailgun.js` + `form-data` add ~12 transitive packages to the dependency tree.

### Mitigation
- `email_status` column on `documents` tracks `PENDING / SENT / FAILED / SKIPPED / DELIVERED / COMPLAINED` — any failure is retryable via `POST /api/documents/email-retry` (batch) or `POST /api/documents/:accessKey/email-retry` (single).
- Email is **fire-and-forget** in `checkAuthorization()` — a send failure never blocks or rolls back the `AUTHORIZED` status update.
- `EMAIL_FAILED` events are written to `document_events` with the error message for audit trail.
- Mailgun's async delivery result (delivered, permanent failure, temporary failure, spam complaint) is captured via `POST /api/mailgun/webhook` (ADR-010), which updates `email_status` and writes the corresponding event. `SENT` therefore means "queued by Mailgun"; `DELIVERED` means confirmed delivery.

### Alternatives Considered
- **nodemailer + SMTP**: More flexible but requires managing an SMTP relay or running a mail server. Deliverability (SPF, DKIM, bounce handling) becomes the application's responsibility. No built-in event log.
- **AWS SES**: Good deliverability and low cost, but adds AWS SDK dependency and IAM credential management. Worth considering if the stack moves to AWS.
- **Storing attachments on disk first**: Rejected — both files are already in memory (DB column + PDF generated by PDFKit) and are small (< 200 KB combined). Disk I/O adds latency and requires cleanup logic with no benefit.
